The Edge-Port Fast Trackdrill

A fresh access port sits in blocking → listening → learning → forwarding — about 30 seconds on classic 802.1D. PCs and phones time out DHCP. PortFast skips listening + learning so the port forwards immediately. BPDU Guard watches for any incoming BPDU and err-disables the port instantly if a rogue switch shows up. What each command does (and why pair them)

spanning-tree portfast

Marks the port as an edge port. Skips listening + learning on link-up — straight to forwarding. The risk: if you accidentally connect another switch here, you bypass the loop-prevention timer that would have caught it. Result = instant L2 loop, broadcast storm, network down.

spanning-tree bpduguard enable

Watches the port for any received BPDU. Real PCs / phones / printers don't send BPDUs — only switches do. If one shows up: port goes err-disabled, traffic stops, syslog. Pair with PortFast and the rogue-switch failure mode is converted from "global outage" to "one dead port and a log line."

Cisco shorthand: set both globally — spanning-tree portfast default + spanning-tree portfast bpduguard default — so every port that becomes access mode automatically gets both. Override on uplinks where appropriate.

Faded prompt — your turn

Helpdesk ticket: "Office floor 3 has DHCP timeouts when users plug in. Once they unplug + replug a few times, it eventually works." What's the fix in two commands?

Answer:

SW1(config)# interface range fa0/1 - 24 SW1(config-if-range)# spanning-tree portfast SW1(config-if-range)# spanning-tree bpduguard enable

PortFast skips ~30s of listening/learning. BPDU Guard catches a rogue switch if one ever lands on these "user" ports.
