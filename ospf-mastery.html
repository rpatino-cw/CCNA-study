Section 01

Overview

OSPF (Open Shortest Path First) is a link-state routing protocol standardized by the IETF (RFC 2328). Every router builds a complete map of the network topology — the Link State Database (LSDB) — and independently runs Dijkstra's SPF algorithm to compute the shortest path to every destination. Because it uses bandwidth-based cost rather than hop count, it makes smarter path decisions. It is classless, supports VLSM/CIDR, and converges faster than distance-vector protocols.

Section 02

Router ID Selection

Every OSPF router needs a unique 32-bit Router ID (RID) — formatted like an IPv4 address. OSPF follows a strict priority order when selecting it:

1
Explicit — router-id command
Manually configured with router ospf 1 → router-id X.X.X.X. This always wins. Does not need to match any interface.
Highest Priority
2
Loopback Interface — Highest IP
If no router-id is configured, OSPF picks the highest IP address among all up loopback interfaces. Loopbacks are preferred because they never go down unless manually shut.
Medium Priority
3
Physical Interface — Highest Active IP
If no loopbacks exist, OSPF uses the highest IP address among all active (up/up) physical interfaces. This RID can change if the interface goes down — which causes OSPF to reconverge.
Lowest Priority
Important
Router ID must be unique per router in the OSPF domain. If you change the Router ID after OSPF is already running, you must run clear ip ospf process (and confirm) for it to take effect. This resets all adjacencies.

Section 04

DR/BDR Election

On multi-access networks (Ethernet), OSPF elects a Designated Router (DR) and a Backup Designated Router (BDR) to reduce the number of OSPF adjacencies and LSDB flooding. All other routers (DROthers) form full adjacencies only with the DR and BDR.

▶ Default priority = 1. Priority 0 = permanently ineligible — that router can never be DR or BDR on that segment.
▶ Highest priority wins. Highest RID is the tiebreaker — not the highest IP address (though they're often the same if using loopbacks).
▶ No preemption: OSPF does NOT preempt. If the DR fails, the BDR takes over. A router joining later with a higher priority does NOT become DR until clear ip ospf process is run.
▶ DR listens on 224.0.0.6 (All OSPF DRs). All routers listen on 224.0.0.5 (All OSPF Routers). DROthers send updates to 224.0.0.6, DR/BDR flood to 224.0.0.5.
Section 05

OSPF Packet Types

Type Name Purpose
1 Hello Neighbor discovery, keepalive, DR/BDR election
2 DBD (Database Description) LSDB summary exchange — used in ExStart and Exchange states
3 LSR (Link State Request) Request specific LSAs from a neighbor during Loading state
4 LSU (Link State Update) Deliver requested LSAs — the actual topology data payload
5 LSAck (Link State Acknowledgment) Confirm receipt of LSU — OSPF uses reliable flooding

Hello Packet Fields

A Hello must contain these fields — and several must match between routers for a neighbor relationship to form:

Router ID Hello interval (must match) Dead interval (must match) Area ID (must match) Authentication (must match) Stub area flag (must match) DR IP address BDR IP address Neighbor list (their RIDs) Subnet mask (P2P: ignored)
Exam Tip
Hello/Dead timers must match for neighbors to form adjacency. Default: Hello = 10s, Dead = 40s (4× Hello). On NBMA/point-to-multipoint: Hello = 30s, Dead = 120s. Mismatched timers = neighbor stuck in Init or Down.
Section 06

LSA Types

LSAs (Link State Advertisements) are the building blocks of the OSPF LSDB. Different LSA types are generated by different router roles and have different flooding scopes.

Type 1

Router LSA

Generated byEvery OSPF router
ScopeStays within the originating area only
ContainsAll of the router's links, costs, and neighbor RIDs
show ip ospf database router
Type 2

Network LSA

Generated byDR only — on multi-access segments
ScopeStays within the area
ContainsList of routers on the segment + network mask
show ip ospf database network
Type 3

Summary LSA

Generated byABR (Area Border Router)
ScopeFlooded between areas
ContainsNetwork prefix + cost summary from one area to another
show ip ospf database summary
Remember
Type 1 and Type 2 LSAs stay within an area. Type 3 crosses area boundaries via ABRs. For the CCNA exam, know these three cold — plus that O IA routes in the routing table come from Type 3 LSAs.

Exam Gotcha
GigabitEthernet and above all cost 1 with the default 100 Mbps reference bandwidth — OSPF literally cannot distinguish between a GigE and a 10GigE link. Fix this with:

router ospf 1
 auto-cost reference-bandwidth 1000

This must be set on ALL routers in the OSPF domain — inconsistent reference bandwidth causes incorrect metric calculations.
Section 08

Configuration Reference

Basic OSPF
R1(config)# router ospf 1 R1(config-router)# network 10.0.0.0 0.0.0.255 area 0 R1(config-router)# network 192.168.1.0 0.0.0.255 area 0

The network command uses a wildcard mask (inverse of subnet mask). It enables OSPF on any interface whose IP falls within that range. The 1 in router ospf 1 is the process ID — local significance only, does not need to match between routers.

Router ID
R1(config)# router ospf 1 R1(config-router)# router-id 1.1.1.1 ! Then reset adjacencies: R1# clear ip ospf process ! → confirm with "yes"
Passive Interface
R1(config-router)# passive-interface GigabitEthernet0/1 ! Or: make ALL passive first, then un-passive the active ones: R1(config-router)# passive-interface default R1(config-router)# no passive-interface GigabitEthernet0/0

Passive interfaces stop sending/receiving Hellos on that interface but still advertise the network in OSPF. Use on LAN segments where no OSPF neighbor exists (e.g., user-facing interfaces).

Reference Bandwidth
R1(config-router)# auto-cost reference-bandwidth 1000 ! Warning: apply to ALL routers in the domain
Interface Cost (Manual Override)
R1(config)# interface GigabitEthernet0/0 R1(config-if)# ip ospf cost 50
OSPF Authentication (MD5)
R1(config)# interface GigabitEthernet0/0 R1(config-if)# ip ospf authentication message-digest R1(config-if)# ip ospf message-digest-key 1 md5 CCNA_PASSWORD

Both sides must use the same key ID and password. MD5 auth is per-interface, not global by default.

Section 09

Troubleshooting Quick Reference

When OSPF adjacency fails, work through these faults in order. Most failures are caused by one of these five issues.

Fault 1 — Area Mismatch
SymptomNeighbor stays Down or Init, never progresses
show cmdshow ip ospf neighbor — empty or missing entry
Root causenetwork statement uses the wrong area number on one side
FixCorrect the area ID in the network command or use ip ospf [pid] area [area] directly on the interface
Fault 2 — Wildcard Mask Error
SymptomNetwork not advertised; routes missing from neighbors
Root causeWrong wildcard mask in network statement — doesn't match the interface IP
FixWildcard = 255.255.255.255 − subnet mask. For /24: mask = .0 → wildcard = 0.0.0.255. For /30: mask = .252 → wildcard = 0.0.0.3
Fault 3 — Hello/Dead Timer Mismatch
SymptomNeighbor flaps (appears and drops) or stays stuck in Init
show cmdshow ip ospf neighbor — neighbor oscillates or Dead Time counts toward 0 and resets
Root causeHello or Dead intervals differ between the two routers — must match exactly
Fixip ospf hello-interval 10 + ip ospf dead-interval 40 — set on both interface sides to match
Fault 4 — Passive Interface on Transit Link
SymptomOne side sees the neighbor, the other doesn't — or adjacency never forms
Root causepassive-interface is configured on a transit (routed) interface — Hellos are suppressed outbound
Fixno passive-interface [interface] on the transit link that connects to a neighbor
Fault 5 — MTU Mismatch (Stuck at ExStart/Exchange)
SymptomNeighbor reaches ExStart or Exchange state but never advances to Full
show cmdshow ip ospf neighbor — State = EXSTART or EXCHANGE (not Full)
Root causeInterface MTU mismatch (e.g., 1500 vs 1476) — DBD packets exceed MTU and are dropped silently
FixMatch MTU: ip mtu 1500 on both sides. Workaround (not best practice): ip ospf mtu-ignore
