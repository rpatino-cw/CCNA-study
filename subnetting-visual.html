Romeo's Strategy — 6 moves to any subnet
Best for /24 – /30 · ≈85% of exam Qs + /16 – /23 · other ~15% →
The big picture: subnetting = splitting one IP block into smaller equal blocks. The magic number is the size of each smaller block. Once you have magic, every other answer falls out — subnet ID, broadcast, hosts, range, next subnet. The whole game is "find magic, then arithmetic."

The whole game: almost every subnet question reduces to "find magic, then arithmetic." Get steps 1 & 2 right (host bits → magic) and 80% of the work is done.
  1. 1 Host bits = 32 − CIDR
    32 = total IPv4 address bits. CIDR is the count of network bits. What's left is the host bits — the digits that name individual machines inside the block.
    /26 → 32 − 26 = 6 host bits
  2. 2 Magic number = 2host bits
    This is the size of each subnet — how many IP addresses it holds, and the step between subnet boundaries. Find this and everything else is just adding/subtracting.
    26 = 64 addresses per subnet
  3. 3 Wildcard = magic − 1
    Same answer as 255 − mask. Used in ACLs and OSPF — it's the inverse of the subnet mask. Goes in the interesting octet; zeros to the left.
    64 − 1 = 63 → 0.0.0.63
  4. 4 Find the interesting octet, then count to the subnet ID
    Interesting octet = the one octet of the mask that isn't 255 or 0. That's the only place subnet boundaries change. For /24–/30 it's octet 4. For /17–/23 it's octet 3. For /9–/15, octet 2. For /1–/7, octet 1.
    In the interesting octet, count multiples of magic from 0 (0, magic, 2×magic, 3×magic…). The largest multiple still ≤ that octet's value is the subnet ID — the first IP of the block (network address, not assignable).
    For 192.168.10.75/26 → octet 4 = 75.
    Count by 64: 0, 64, 128, 192.   75 sits between 64 and 128.
    → Subnet ID = 192.168.10.64
  5. 5 Broadcast = subnet ID + magic − 1
    Equivalently: next subnet ID − 1. Last IP of the block. Reserved for broadcast — also not assignable.
    64 + 64 − 1 = 192.168.10.127
  6. 6 First / last usable, count, next subnet
    Once you have subnet ID + broadcast, the rest is one-line arithmetic. 
    first usable = subnet ID + 1     → 192.168.10.65
last usable  = broadcast − 1     → 192.168.10.126
usable hosts = magic − 2         → 62
next subnet  = subnet ID + magic → 192.168.10.128

Worked exam questions — easy → hard
Mini-example 1 — "What is the subnet mask for shorthand /11?"
Foundation question — learn this first. "Shorthand" = CIDR notation. /N just means "the mask has N ones at the front, then zeros." Same info as dotted-decimal, fewer keystrokes. /24 = 255.255.255.0.
/11 = 11 ones, 21 zeros. Split across 4 octets:
octet 1 (bits 1–8):  11111111 = 255
octet 2 (bits 9–11):  11100000 = 224  (3 ones, 5 zeros)
octet 3:  00000000 = 0
octet 4:  00000000 = 0
→ mask = 255.224.0.0
Faster shortcut: figure out which octet the CIDR lives in. /1–/8 → octet 1 · /9–/16 → octet 2 · /17–/24 → octet 3 · /25–/32 → octet 4. Octets to the left = 255. Octets to the right = 0. The interesting octet gets the partial mask. /11 lives in octet 2 → 255.X.0.0 where X is the partial. Look up X in the chart's "Mask" column for /11 → 224.
Mini-example 2 — "Enter the first valid host on 172.20.98.0 with mask 255.255.255.0"
First step always: translate dotted-decimal masks into CIDR. 255.255.255.0 = /24 (24 leading 1-bits). So the network is 172.20.98.0/24.
host bits = 32 − 24 = 8   → magic = 28 = 256 (whole octet 4)
subnet ID = 172.20.98.0   (already given)
broadcast = 0 + 256 − 1 = 172.20.98.255
first valid host = subnet ID + 1 = 172.20.98.1 ✓
The shortcut: "first valid host" is always subnet ID + 1. Subnet ID itself = network address (reserved). The +1 is the first IP a real device can use. "Last valid host" = broadcast − 1.
Mini-example 3 — "What subnet does host 172.27.25.250/16 belong to?"
The trick: /16 is a clean octet boundary (Class B). When the CIDR lands exactly on /8, /16, or /24, you don't even need magic — just split the IP into "network octets" and "host octets" by counting bits.
/16 = 16 network bits = 2 full octets of network.
IP:  172.27.25.250
      ↑ network   ↑ host bits
Keep network octets, zero out host octets:
→ subnet ID = 172.27.0.0 ✓
The pattern for clean boundaries: /8 → keep octet 1, zero rest  ·  /16 → keep octets 1–2, zero rest  ·  /24 → keep octets 1–3, zero octet 4.
For non-clean CIDRs (anything that isn't a multiple of 8), you go back to the magic-number method on the interesting octet. The full strategy still works, this is just the shortcut for the easy cases.
Verify with Romeo's Strategy if you want: host bits = 32 − 16 = 16 → magic = 216 = 65,536. 65,536 ÷ 256 = 256 (whole octet 3). 256 ÷ 256 = 1 → step by 1 in octet 2 (every value of octet 2 is its own /16). 172.27.x.x → subnet ID 172.27.0.0. ✓
Mini-example 4 — "are these two IPs on the same subnet?" (CCNA's favorite trap)
The technique: run Romeo's Strategy on each IP independently, then compare subnet IDs. Same ID = same subnet. Different ID = different subnet. The trap is non-clean CIDRs where two IPs in the "same /24" can land in different /26's.
Question: Are 192.168.10.50/26 and 192.168.10.100/26 on the same subnet?
Magic for /26 = 64.   Count by 64 in octet 4: 0, 64, 128, 192.
.50 → between 0 and 64  →  subnet ID 192.168.10.0
.100 → between 64 and 128  →  subnet ID 192.168.10.64
Different subnet IDs → NOT same subnet
Why this trips people up: both IPs share octets 1–3 ("192.168.10") so they look like neighbors. But /26 cuts each /24 into 4 pieces — the boundary between piece 1 (.0–.63) and piece 2 (.64–.127) lands right between .50 and .100.
Mini-example 5 — "First valid host on the network for 172.29.40.202 with mask 255.255.252.0"
This one's harder than 2. The mask isn't a clean boundary, so you can't just zero an octet — magic lands in octet 3. Two-step play: translate the mask to CIDR, then run the full Romeo's Strategy.
Step 0 — translate the mask:
255.255.252.0 → octet 3 = 252 = 11111100 (6 ones)
Total ones = 8 + 8 + 6 = /22

Steps 1–2 — find magic:
host bits = 32 − 22 = 10 → magic = 210 = 1,024
1,024 > 256 → push left. 1,024 ÷ 256 = 4 → step by 4 in octet 3

Step 4 — find subnet ID:
IP octet 3 = 40. Count by 4: 0, 4, 8, … 36, 40, 44…
40 lands exactly on a boundary → subnet ID = 172.29.40.0

Step 5 — find broadcast:
next subnet ID = 172.29.44.0 (40 + 4 in octet 3)
broadcast = next − 1 = 172.29.43.255

Step 6 — first valid host:
subnet ID + 1 = 172.29.40.1 ✓
Why the broadcast jumps to .43.255: when the magic lives in octet 3, the block spans multiple octet-4 cycles. /22 contains 4 full /24 worth of addresses (4 × 256 = 1,024). So the block 172.29.40.0/22 covers .40.0 → .40.255 → .41.0 → .41.255 → .42.0 → .42.255 → .43.0 → .43.255. First valid host = .40.1, last valid host = .43.254.
Sanity check: usable hosts = magic − 2 = 1,024 − 2 = 1,022. Cross-reference the chart row for /22 → "Hosts" column or the universal formula 210 − 2 = 1,022. ✓
Mini-example 6 — "Split 172.16.0.0/16 into 16 equal subnets — max usable hosts per subnet?"
This is a "borrow bits" question. Different shape: instead of given CIDR → find hosts, you're given a parent block + how many subnets you want → find the new CIDR → then find hosts. Three moves.
Move 1 — how many bits to borrow:
Need 16 subnets. Find smallest N where 2N ≥ 16.
24 = 16 ✓   → borrow 4 bits

Move 2 — new CIDR:
parent /16 + 4 borrowed = /20   (every subnet is now /20)

Move 3 — usable hosts per subnet:
host bits = 32 − 20 = 12   → 212 − 2 = 4,094 usable hosts ✓
The pattern (memorize this):
1. Subnets needed → find N: smallest 2N ≥ that number
2. New CIDR = parent CIDR + N
3. Hosts per subnet = 2(32 − new CIDR) − 2
Sanity check by listing the 16 subnets: magic for /20 = 16 in octet 3. So subnets step by 16 in octet 3:
172.16.0.0/20 · 172.16.16.0/20 · 172.16.32.0/20 · 172.16.48.0/20 · … 172.16.240.0/20 (16 subnets total, last one ends at 172.16.255.255). Each holds 4,094 usable. ✓
Watch for: if the question said "32 subnets," you'd borrow 5 bits (25=32) → /21 → 211−2 = 2,046 hosts each. If it said "20 subnets," still borrow 5 (because 24=16 isn't enough) → /21 again → 32 subnets, 12 unused. Round up to the next power of 2 every time.
Mini-example 7 — "Smallest prefix for 200 hosts?"
This is VLSM in reverse — host count first, find the CIDR. Where mini-example 6 asked "how many hosts fit," this asks "what's the smallest block that fits N hosts." Three moves, opposite direction.
Move 1 — find h, the smallest host-bit count that fits 200:
test powers of 2 (minus 2 for network/broadcast):
  27 − 2 = 126   ← too small
  28 − 2 = 254   ← ≥ 200 ✓ → h = 8

Move 2 — convert to CIDR:
CIDR = 32 − h = 32 − 8 = /24 ✓

Sanity check: /24 fits 254 usable hosts. Need 200. Plenty of room. /25 would only fit 126 — not enough.
The whiteboard trick: memorize the powers-of-2 ladder once, the rest is lookup.

Common gotcha: always round up. 200 hosts ≠ /25 just because it's "close to 126" — the exam wants fits all hosts, not almost fits. /25 leaves 74 hosts homeless. Always pick the row where usable ≥ requested.
Heads up — for /17 to /23, the magic moves to a different octet.
Magic from Step 2 = total block size (total addresses per subnet). For /24–/30 that block fits inside one octet (max 256), so magic IS the step in octet 4 — easy mode. For /17–/23, the block is bigger than 256, so it can't fit in the 4th octet. It bumps up to the 3rd octet, and octet 4 just runs 0–255 inside each block.
/20 walkthrough — for IP 192.168.10.5/20:
host bits = 32 − 20 = 12 → magic = 212 = 4,096 (total block size)
4,096 ÷ 256 = 16 → step by 16 in the 3rd octet
interesting octet = 3 (octet 3 of the IP is "10")
count by 16 in octet 3: 0, 0, 16, 32… 10 sits between 0 and 16 → subnet ID 192.168.0.0
broadcast = 192.168.15.255   ·   usable .0.1 – .15.254
Shortcut rule: if your magic is > 256, divide by 256 — that's the real step, applied to the octet one to the left.
Edge cases the exam loves:
/31 = 2 usable hosts on point-to-point links (RFC 3021). Don't apply magic − 2 here — it'd give 0.
/32 = 1 host route (loopback, DNS A record, route filter). No subnetting at all.

Class boundaries: /8 (A), /16 (B), /24 (C) — and why /24 is the middle man
Three class boundaries split the CIDR space, one per octet. Each marks where one octet of network bits "fills up" and the next octet starts.
A · /8
1st octet full · ~16M hosts · ISP-scale
B · /16
2 octets full · ~65K hosts · large enterprise
★ C · /24
3 octets full · 254 hosts · default LAN
/24 is the middle man — the CCNA exam phrases ~80% of subnetting questions as "given a /24, split into..." So you start from /24 and borrow bits to make /25, /26, /27, etc.
  • /1 – /23: Bigger than /24 (route summarization, ISP-side, classful aggregation)
  • /24: ⭐ Class C boundary, default LAN, full octet of host space
  • /25 – /32: Smaller than /24 (subnetted — Hosts + Block boundaries columns become valid here)
In the table: /24 is the bridge — every /25–/32 is "carved out of a /24," every /17–/23 is "/24s grouped together," and /1–/16 climb further up the class ladder.
Wait — does Class A/B/C apply to the whole column or just the boundary?
Class A/B/C labels apply to the specific boundary, not the whole column. The classes mark where each octet of network bits "fills up":
  • /8 = Class A boundary — 1st octet full of network bits
  • /16 = Class B boundary — first 2 octets full
  • /24 = Class C boundary — first 3 octets full
Bigger than a class boundary = aggregation. (e.g. /20 is bigger than /24, sits between Class B and C.)
Smaller than a class boundary = subnetted version of that class. (e.g. /27 is a subnetted /24, still "Class C space" if the parent block was Class C.)

Exam reality: classes are legacy — CIDR replaced classful in 1993 (RFC 1518). On the CCNA, "Class C" really just means "/24-sized network." The boundaries still get tested because they shape RFC 1918 ranges and default route summarization behavior.
The CIDR columns themselves just group prefixes that share a bit pattern in the interesting octet — class is about the boundary, not the whole column.
How do I find # hosts for /1–/24? (Hosts column shows /25–/32 only)
Universal formula — works for any CIDR:
# hosts = 2(32 − CIDR) − 2
The −2 = subnet ID + broadcast address (always reserved). Edge cases: /31 = 2 hosts (P2P, RFC 3021), /32 = 1 host (host route, no reservation).

Hosts = 131,072 − 2 = 131,070

Your factorization angle: 217 = 128 × 1024 = 131,072. Same answer, different path. Powers of 2 always factor cleanly because exponents add.
Bottom line: the math is straightforward, just yields huge numbers. Useful for CCNP/Cisco design exams, not CCNA. If a CCNA question gives you a /15 host count, the answer choices will differ by orders of magnitude — pick ">100,000" not "131,070."

How to answer any question with this table
Q: "How many usable hosts on a /27?"
Find /27 → group size = 32 → usable = 32 - 2 = 30 hosts. Mask = 255.255.255.224.

Q: "What subnet is 192.168.1.47/28 in?"
Find /28 → group size = 16 → count by 16: 0, 16, 32, 48... 47 falls in the 32 block → network = 192.168.1.32, broadcast = 192.168.1.47.

Q: "What is the broadcast for 10.0.0.130/25?"
Find /25 → group size = 128 → blocks: 0, 128, 256... 130 is in the 128 block → broadcast = 128 + 128 - 1 = 10.0.0.255.

Two buckets — what each half means
🏢

Net Bits — the building

The network portion. Tells routers which network a packet is destined for. Every device on the same LAN shares these bits exactly — they all live in the same building.

"Same net bits = same network = no router needed"
🚪

Host Bits — the room number

The host portion. Identifies which device on that network. Each device on the LAN must have a unique value here. Two reserved values: all-zeros (network) and all-ones (broadcast).

"Host bits = 2h − 2 usable addresses"

The mask's 1s lock in the network bits. Its 0s let the host bits float. Click below to watch the AND fill in.

4. The Long Way — Binary AND Method

Textbook method. Slow but essential to understand. Click through each step.

Example: Find the subnet for 192.168.10.75/26
1

Convert the IP to binary

192 = 11000000   168 = 10101000   10 = 00001010   75 = 01001011
2

Write the /26 mask in binary

11111111.11111111.11111111.11000000

26 ones, then 6 zeros. Only the last octet is "interesting" (not all 1s or all 0s).

3

AND the last octet (the interesting one)

IP:   01001011 (75)
Mask: 11000000 (192)
AND:  01000000 (64)
4

Results

Network: .64 First host: .65 Last host: .126 Broadcast: .127

62 usable hosts. This took 2-3 minutes. The magic number method does it in 10 seconds.

8. Route Summarization — Combine Routes Into One

Summarization (supernetting) is the reverse of subnetting: combine multiple specific routes into one advertisement. Find the common prefix bits.

Live Summarizer — Enter networks to combine

Comma-separated. Example: 192.168.16.0/24, 192.168.17.0/24, 192.168.18.0/24, 192.168.19.0/24

The Method
  1. Convert all network addresses to binary
  2. Line them up and compare bit-by-bit from the left
  3. Count how many bits are identical — that's your summary prefix length
  4. The summary route is the common bits + all zeros for the rest

9. Wildcard Masks — Type a Mask, See the Inverse

ACLs and OSPF use wildcard masks. Just subtract each octet from 255.

What is a wildcard? — The bit inversion

A wildcard mask is the bitwise NOT of a subnet mask. Every 1 becomes 0, every 0 becomes 1. Click "Flip" and watch.

Subnet mask
255.255.255.192
↓  F L I P  ↓
Wildcard
0.0.0.63

0 = match exactly (Cisco semantic). 1 = don't care. So a wildcard of 0.0.0.255 means "match the first 3 octets exactly, ignore the last one."

Why two masks? — Three jobs, two shapes
Subnet mask
Defines a network

Used on interfaces. Tells routers "all addresses with these network bits live on the same LAN."

Shape: contiguous 1s from the left. Always.

ip address 10.1.1.1 255.255.255.0
Wildcard (ACL)
Match-rule pattern

Used in access lists. Selects which IPs to permit or deny.

Shape: bits can be non-contiguous (e.g. match every odd host).

access-list 10 permit 10.1.1.0 0.0.0.255
Wildcard (OSPF)
Network statement

Tells OSPF which interfaces to advertise into an area.

Same syntax as ACL wildcard. 0 = match, 1 = ignore.

network 10.1.1.0 0.0.0.255 area 0

11. Exam Traps — Click to Reveal

Cisco writes wrong answers that look right. Click each trap to see how they trick you.

Trap #1 — Forgetting to subtract 2▼
WRONG: /26 = 64 hosts
RIGHT: /26 = 64 - 2 = 62 usable hosts
Network address and broadcast address are NOT assignable to hosts. Always subtract 2.
Trap #2 — IP is the network or broadcast address▼
"Can 192.168.1.64 be assigned to a host on 192.168.1.64/26?"
NO — .64 IS the network address. First usable = .65. Same with broadcast: .127 is NOT usable.
Trap #3 — Two hosts on different subnets▼
"Can 10.0.0.62 and 10.0.0.65 communicate on a /26?"
.62 is in subnet .0-.63  |  .65 is in subnet .64-.127
Different subnets — need a router. Use the magic number to quickly find which subnet each IP belongs to.
Trap #4 — /31 and /32 edge cases▼
/31 = 2 hosts, both usable (RFC 3021, point-to-point only, no broadcast)
/32 = single host route (routing tables and loopbacks, NOT subnetting)
Trap #5 — Mask doesn't fit what you expect▼
"Subnet mask for 200 hosts?"
/24 = 254 hosts (enough!)  |  /25 = only 126 (NOT enough!)
Answer: 255.255.255.0 (/24) — Always pick the smallest prefix that fits.
