One of the benefits of a layered system like&nbsp;

Well, in our example, our server, SRV1, provides&nbsp;&nbsp;

UDP, User Datagram Protocol, and TCP, Transmission&nbsp;&nbsp;
Control Protocol. They both offer some different&nbsp;
features, and we’ll cover them in this course.
Finally, we have layer 5: the Application&nbsp;
layer. This is where network communications&nbsp;&nbsp;
meet applications. Small side note:&nbsp;
this layer is usually called “Layer&nbsp;&nbsp;
7”. I’ll explain why later when we talk about&nbsp;
another model, the OSI model. The Application&nbsp;&nbsp;
layer defines how application processes&nbsp;
format, send, and interpret data. So,&nbsp;&nbsp;
when Chrome on PC1 sends a request&nbsp;
to the web server process on SRV1,&nbsp;&nbsp;
it uses an Application-layer protocol such&nbsp;
as HTTP to format and send the message,&nbsp;&nbsp;
and that same protocol tells the web server how&nbsp;
to interpret the message it receives. Protocols&nbsp;&nbsp;
at this layer define message formats and rules&nbsp;
for specific tasks. Here are a few examples. For&nbsp;&nbsp;
browsing web pages, we use HTTP, or more often&nbsp;
its secure version, HTTPS. For file transfers,&nbsp;&nbsp;
we can use FTP, TFTP, or another similar&nbsp;
protocol. And email uses its own protocols, too.&nbsp;&nbsp;
Don’t worry about the specific protocols or their&nbsp;
names at this point, right now we’re just talking&nbsp;&nbsp;
big concepts. Network infrastructure devices&nbsp;
like routers and switches typically don’t care&nbsp;&nbsp;
about Application-layer details. This is too&nbsp;
high-level for them. They just move messages&nbsp;&nbsp;
across the network. Only the communicating&nbsp;
hosts, PC1 and SRV1 in this case, actually&nbsp;&nbsp;
look at and interpret the Application-layer&nbsp;
data. Now that we’ve seen what each layer does,&nbsp;&nbsp;
how does a single message actually include all of&nbsp;
this information at once? Let’s see how the layers&nbsp;&nbsp;
combine into a stack and how data moves through&nbsp;
them using encapsulation and decapsulation.
To show how encapsulation and decapsulation&nbsp;
work, let’s simplify the network, making it a&nbsp;&nbsp;
direct connection between PC1 and SRV1. First,&nbsp;
the Application layer prepares the data to be&nbsp;&nbsp;
sent over the network, for example an HTTP request&nbsp;
that Chrome on PC1 sends to a web server running&nbsp;&nbsp;
on SRV1. As the message moves down the stack, each&nbsp;
layer encapsulates the data with a header that&nbsp;&nbsp;
contains the information needed by that layer.&nbsp;
For example, the source and destination addresses:&nbsp;&nbsp;
Layer 4 port numbers, layer 3 IP addresses, and&nbsp;
Layer 2 MAC addresses, among other information.&nbsp;&nbsp;
So first, the Transport layer encapsulates&nbsp;
the data with its header, the Layer 4 header,&nbsp;&nbsp;
with source and destination port numbers&nbsp;
and other information. Then the Internet&nbsp;&nbsp;
layer adds its header with source and destination&nbsp;
IP addresses. Then, Layer 2 encapsulates the data&nbsp;&nbsp;
with both a header and a trailer. The trailer&nbsp;
is used by the receiving device to check for&nbsp;&nbsp;
transmission errors. We’ll look at this in&nbsp;
more detail in a later video about Ethernet.&nbsp;&nbsp;
Finally, the Physical layer transmits the&nbsp;
bits as signals over the physical medium,&nbsp;&nbsp;
which is an Ethernet cable in this case. I’m&nbsp;
showing the message in order from left to&nbsp;&nbsp;
right here. So the Layer 2 header is transmitted&nbsp;
first, and Layer 2 trailer is transmitted last.
So that was the encapsulation process, now let’s&nbsp;
look at decapsulation. The receiving device&nbsp;&nbsp;
receives the message as a stream of bits at layer&nbsp;
1. Layer 1 simply passes those bits up to the next&nbsp;&nbsp;
layer. Now it’s up to the other layers to&nbsp;
interpret those bits. The device examines the&nbsp;&nbsp;
information in the Layer 2 header and trailer, and&nbsp;
then removes them. This is called “decapsulation”,&nbsp;&nbsp;
or sometimes “de-encapsulation”. The decapsulation&nbsp;
process continues up the stack. So, Layer&nbsp;&nbsp;
3 examines and removes the Layer 3 header. Then&nbsp;
Layer 4 examines and removes the Layer 4 header,&nbsp;&nbsp;
and the data is delivered to the Application&nbsp;
layer. Now the application processes the data and,&nbsp;&nbsp;
if needed, generates a response that goes back&nbsp;
down the stack, and is then transmitted back to&nbsp;&nbsp;
PC1. So, that’s the decapsulation process, which&nbsp;
is basically the encapsulation process in reverse.
Each device has its own network stack.&nbsp;
A message from PC1 goes down its stack,&nbsp;&nbsp;
crosses the wire, and then goes up SRV1’s&nbsp;
stack where the Application layer processes&nbsp;&nbsp;
it. The response does the same thing in the&nbsp;
opposite direction. It goes down SRV1’s stack,&nbsp;&nbsp;
is sent over the wire to PC1, and then goes&nbsp;
up PC1’s stack where the data is interpreted&nbsp;&nbsp;
by the application. In a real network there are&nbsp;
switches, routers, and other devices between the&nbsp;&nbsp;
hosts. That adds more steps to the delivery&nbsp;
process, just like the mail system example,&nbsp;&nbsp;
where the letter is moved between cars and trucks.&nbsp;
But the overall flow is still the same: each&nbsp;&nbsp;
host sends the message down its stack, across the&nbsp;
network, and up the other host’s stack. We’ll look&nbsp;&nbsp;
at how those intermediate devices - the routers&nbsp;
and switches - handle messages in later videos.
Now that we’ve seen how encapsulation and&nbsp;
decapsulation work, let’s look at what to call&nbsp;&nbsp;
these messages. At each stage in the encapsulation&nbsp;
and decapsulation process, there is a name given&nbsp;&nbsp;
to the message. The combination of data and a&nbsp;
Layer 4 header is called a segment when using TCP,&nbsp;&nbsp;
or a datagram when using UDP. Remember that point:&nbsp;
TCP creates segments, and UDP creates datagrams.&nbsp;&nbsp;
This difference has to do with how TCP and UDP&nbsp;
treat the data, but we don’t need those details&nbsp;&nbsp;
yet. The combination of a segment or datagram and&nbsp;
a Layer 3 header is called a packet. “Packet” is&nbsp;&nbsp;
the most common term we use when talking about&nbsp;
messages being sent over a network, but strictly&nbsp;&nbsp;
speaking it refers to the message at this stage in&nbsp;
the process. Finally, the combination of a packet&nbsp;&nbsp;
and a Layer 2 header and trailer is called&nbsp;
a frame. This is what is actually sent over&nbsp;&nbsp;
the wire. You’ll never see a packet, segment,&nbsp;
or datagram travelling over the wire itself;&nbsp;&nbsp;
they are always sent inside a frame. We can also&nbsp;
use alternative names to describe the message at&nbsp;&nbsp;
each stage, using the term protocol data unit,&nbsp;
or PDU. A segment or datagram is a Layer 4 PDU,&nbsp;&nbsp;
or L4PDU. A packet is a Layer 3 PDU, or L3PDU. And&nbsp;
a frame is a Layer 2 PDU, or L2PDU. Both names are&nbsp;&nbsp;
common, so I recommend knowing both. And there’s&nbsp;
one more important term to learn. The contents&nbsp;&nbsp;
of each PDU, that is everything encapsulated by&nbsp;
the layer’s header and trailer, are called the&nbsp;&nbsp;
payload. So, at Layer 4 a segment or datagram’s&nbsp;
payload is the application data itself. At Layer&nbsp;&nbsp;
3, a packet’s payload is a segment or datagram,&nbsp;
including the Layer 4 header and the data. And at&nbsp;&nbsp;
Layer 2, a frame’s payload is a packet, including&nbsp;
the Layer 3 and Layer 4 headers and the data. Just&nbsp;&nbsp;
remember that the payload is what’s inside the&nbsp;
PDU, not including that layer’s header or trailer.
Finally, let’s talk about how these layers&nbsp;
interact with each other. I’ve replaced the&nbsp;&nbsp;
generic layer names with examples of common&nbsp;
protocols. Also note the distinction between&nbsp;&nbsp;
Layer 2 Ethernet, which defines MAC addressing&nbsp;
and frames, and Ethernet at the physical layer,&nbsp;&nbsp;
which defines the signaling and physical&nbsp;
media. Each layer in the stack provides a&nbsp;&nbsp;
service to the layer above it, and is serviced by&nbsp;
the layer below it. This is called adjacent-layer&nbsp;&nbsp;
interaction. For example, Layer 4 provides a&nbsp;
service to Layer 5 by delivering data to the&nbsp;&nbsp;
correct application using port numbers. Layer&nbsp;
3 provides a service to Layer 4 by delivering&nbsp;&nbsp;
segments and datagrams to the correct destination&nbsp;
host using IP addresses. Layer 2 provides a&nbsp;&nbsp;
service to Layer 3 by delivering packets to&nbsp;
the next hop using MAC addresses. And Layer&nbsp;&nbsp;
1 provides a service to Layer 2 by sending and&nbsp;
receiving the frame’s bits as electrical, optical,&nbsp;&nbsp;
or radio signals over the physical medium. Each&nbsp;
layer relies on the layer below it to do its job.
There’s also a related concept. Each layer&nbsp;
communicates with the same layer on other devices,&nbsp;&nbsp;
and this is called same-layer interaction.&nbsp;
For example, the Application layer on one host&nbsp;&nbsp;
sends data to the Application layer on the other&nbsp;
host. A segment or datagram is addressed to the&nbsp;&nbsp;
Layer 4 port number of the correct application&nbsp;
on the destination host. A packet is addressed&nbsp;&nbsp;
to the Layer 3 IP address of the destination&nbsp;
host. A frame is addressed to the Layer 2 MAC&nbsp;&nbsp;
address of the next hop. And signals sent&nbsp;
out of a physical port are received by a&nbsp;&nbsp;
physical port on the connected device.&nbsp;
This layered cooperation - within each&nbsp;&nbsp;
device and between devices - is what&nbsp;
makes network communication possible.
Just like in our mail system example, the&nbsp;
separation of layers is key to how networks&nbsp;&nbsp;
work. Each layer has its own job and provides a&nbsp;
specific service to the layers above. Because the&nbsp;&nbsp;
layers are modular, we can swap protocols at one&nbsp;
layer without changing the others. For example,&nbsp;&nbsp;
instead of a web page exchange using HTTP and&nbsp;
TCP, maybe it’s a file exchange using TFTP,&nbsp;&nbsp;
Trivial File Transfer Protocol, over UDP.&nbsp;
The lower layers in the stack can use IP&nbsp;&nbsp;
and Ethernet without worrying about&nbsp;
the details of what the upper layers&nbsp;&nbsp;
are doing. And if instead of a wired Ethernet&nbsp;
connection it’s a wireless Wi-Fi connection,&nbsp;&nbsp;
Layers 1 and 2 can change to Wi-Fi without&nbsp;
affecting the upper layers. As long as each&nbsp;&nbsp;
layer keeps its “contract” with the layers above&nbsp;
and below, we can improve or replace protocols at&nbsp;&nbsp;
different layers without redesigning everything.&nbsp;
That flexibility is one of the main benefits of a&nbsp;&nbsp;
layered model. Now that we’ve built&nbsp;
this 5-layer TCP/IP model, let’s see&nbsp;&nbsp;
how it compares to other models you’ll see in&nbsp;
other courses and books, like the OSI model.
You might have heard of the OSI model&nbsp;
before. If you’ve studied IT before,&nbsp;&nbsp;
you almost certainly have. As I said at the&nbsp;
beginning of this video, TCP/IP development&nbsp;&nbsp;
started in the 1970s, with work on ARPANET and&nbsp;
the early TCP/IP specifications. But in the late&nbsp;&nbsp;
1970s and 1980s, the International Organization&nbsp;
for Standardization, the ISO, designed a 7-layer&nbsp;&nbsp;
model called OSI, Open Systems Interconnection,&nbsp;
and a matching protocol suite. The goal was to&nbsp;&nbsp;
create international, vendor-neutral networking&nbsp;
standards that could unify existing proprietary&nbsp;&nbsp;
stacks and potentially replace TCP/IP as the&nbsp;
vendor-neutral choice. Here is that model,&nbsp;&nbsp;
with seven layers from top to bottom: Application,&nbsp;
Presentation, Session, Transport, Network, Data&nbsp;&nbsp;
Link, and Physical. Governments, including the&nbsp;
US, promoted OSI as the preferred or recommended&nbsp;&nbsp;
stack for new deployments. And for a while many&nbsp;
people thought that OSI was the future. But OSI&nbsp;&nbsp;
protocols ended up being developed too late and&nbsp;
were too complex, and they never gained the same&nbsp;&nbsp;
level of deployment as TCP/IP. I wasn’t around&nbsp;
at the time, but my understanding is that this&nbsp;&nbsp;
was largely due to the bureaucratic nature of&nbsp;
the approach to developing OSI and its protocol&nbsp;&nbsp;
stack. The process was very top-down: committees&nbsp;
designed the protocols in detail, and vendors were&nbsp;&nbsp;
expected to implement them exactly as specified.&nbsp;
TCP/IP used a more bottom-up approach. In the end,&nbsp;&nbsp;
TCP/IP won in the real world, although some&nbsp;
OSI technologies are still used. Today,&nbsp;&nbsp;
almost all networks use the TCP/IP protocol stack,&nbsp;
but the 7-layer OSI model survives as a reference&nbsp;&nbsp;
and teaching model, and as a common way to talk&nbsp;
about layers. Actually, most networking resources&nbsp;&nbsp;
use a 5-layer model like the one I covered in this&nbsp;
video, but with names borrowed from the OSI model.&nbsp;&nbsp;
Here is that 5-layer model. I foreshadowed this&nbsp;
earlier in the video, but this is why the TCP/IP&nbsp;&nbsp;
Application layer is often called Layer 7: because&nbsp;
the Application layer is Layer 7 of the OSI model.&nbsp;&nbsp;
I’ve also highlighted the two layer names that&nbsp;
are different from the model I introduced:&nbsp;&nbsp;
Layer 3 is called the Network layer, and&nbsp;
Layer 2 is called the Data Link layer.
