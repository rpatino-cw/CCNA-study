Okay, we’ve covered BPDU Guard, now let’s move on&nbsp;
to BPDU Filter. Before we look at how it works,&nbsp;&nbsp;
let’s see the problem it solves. A switch port&nbsp;
connected to an end host continues sending BPDUs&nbsp;&nbsp;
every 2 seconds. And it does this regardless of&nbsp;
whether PortFast and BPDU Guard are enabled. So,&nbsp;&nbsp;
even though Bob’s PC doesn’t run spanning&nbsp;
tree and simply ignores any BPDUs,&nbsp;&nbsp;
SW3 keeps sending BPDUs to Bob’s PC, once every 2&nbsp;
seconds. If the port doesn’t connect to a switch,&nbsp;&nbsp;
sending BPDUs is both unnecessary and also&nbsp;
undesirable, for a couple of reasons. First&nbsp;&nbsp;
is that sending BDPUs uses some bandwidth and&nbsp;
processing power on the switch, although it’s&nbsp;&nbsp;
minimal. Not a big deal, but still unnecessary.&nbsp;
Second is that BPDUs contain information about&nbsp;&nbsp;
the LAN’s STP topology. And if maximum security&nbsp;
is a concern, you should avoid sending this info&nbsp;&nbsp;
to user devices. BPDU Filter solves this&nbsp;
by preventing a port from sending BPDUs.
Let’s explore how BPDU Filter works. As I just&nbsp;
mentioned it stops a port from sending BPDUs. So,&nbsp;&nbsp;
if you enable it on SW3 G0/1, it won’t send any&nbsp;
more BPDUs to Bob’s PC. But unlike BPDU Guard,&nbsp;&nbsp;
BPDU Filter doesn’t disable the port if it&nbsp;
receives a BPDU. How the switch reacts when it&nbsp;&nbsp;
receives a BPDU depends on how you configure BPDU&nbsp;
Filter, so let’s see how to do that. BPDU Filter&nbsp;&nbsp;
can be enabled in two ways, just like PortFast&nbsp;
and BPDU Guard: it can be configured per-port&nbsp;&nbsp;
in interface config mode, or by default in&nbsp;
global config mode. However, unlike PortFast&nbsp;&nbsp;
and BPDU Guard, BPDU Filter functions differently&nbsp;
depending on how you configure it. To configure it&nbsp;&nbsp;
on a specific port, use SPANNING-TREE BPDUFILTER&nbsp;
ENABLE in interface config mode. With BPDU Filter&nbsp;&nbsp;
enabled, the port will not send BPDUs. On top of&nbsp;
that, the port will ignore any BPDUs it receives.&nbsp;&nbsp;
Basically, this disables STP on the port, and&nbsp;
for that reason you should use it with caution,&nbsp;&nbsp;
in bold red letters. PortFast enabled on the wrong&nbsp;
port can cause temporary layer-2 loops. But BPDU&nbsp;&nbsp;
Filter enabled on the wrong port can cause&nbsp;
permanent layer-2 loops, and is a great way&nbsp;&nbsp;
to cause a broadcast storm and bring down the&nbsp;
network. STP is a key protocol for preventing&nbsp;&nbsp;
loops, so disabling it on a port can be very&nbsp;
risky. A better choice is to enable BPDU Filter&nbsp;&nbsp;
by default in global config mode. The command is&nbsp;
SPANNING-TREE PORTFAST BPDUFILTER DEFAULT. And&nbsp;&nbsp;
again, you can optionally use the EDGE keyword&nbsp;
after PortFast. Just like when enabling BPDU&nbsp;&nbsp;
Guard by default, BPDU Filter will be activated&nbsp;
on all PortFast-enabled ports. And then you can&nbsp;&nbsp;
use SPANNING-TREE BPDUFILTER DISABLE to disable it&nbsp;
on specific ports if necessary. When BPDU Filter&nbsp;&nbsp;
is activated on a port by default, the port will&nbsp;
not send BPDUs, just like above. But here’s the&nbsp;&nbsp;
difference. If the port receives a BPDU, PortFast&nbsp;
and BPDU Filter are disabled, and it operates as a&nbsp;&nbsp;
normal STP port. This is the best of both worlds.&nbsp;
Under normal circumstances the port doesn’t send&nbsp;&nbsp;
BPDUs, but if it receives BPDUs it is able to&nbsp;
react appropriately – it doesn’t just ignore them.
Before we finish up, let me give you my&nbsp;
recommendation and one final note about&nbsp;&nbsp;
BPDU Guard and BPDU Filter. You can enable&nbsp;
PortFast and BPDU Guard however you prefer,&nbsp;&nbsp;
either per-port or by default. It doesn’t affect&nbsp;
