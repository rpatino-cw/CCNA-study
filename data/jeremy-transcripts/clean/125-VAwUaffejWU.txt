So, I mentioned that Ansible, Puppet, and Chef are&nbsp;
primarily configuration management tools, whereas&nbsp;&nbsp;
Terraform is primarily a provisioning tool. Let’s&nbsp;
clarify what the difference is. First, what is&nbsp;&nbsp;
configuration management? Tools like Ansible,&nbsp;
Puppet, and Chef manage existing infrastructure&nbsp;&nbsp;
by installing software, configuring settings, and&nbsp;
maintaining system state. This ensures consistency&nbsp;&nbsp;
by applying and enforcing configurations across&nbsp;
multiple devices. Again, these are the benefits&nbsp;&nbsp;
of Ansible, Puppet, and Chef that I covered&nbsp;
in the previous video. As for infrastructure&nbsp;&nbsp;
provisioning, it creates, modifies, and deletes&nbsp;
infrastructure resources such as servers and&nbsp;&nbsp;
network infrastructure. For example, creating&nbsp;
virtual servers and routers on a cloud platform&nbsp;&nbsp;
like AWS. But the focus is on initial setup&nbsp;
rather than ongoing configuration management.&nbsp;&nbsp;
To summarize, configuration management tools work&nbsp;
on already existing systems, whereas provisioning&nbsp;&nbsp;
tools build infrastructure from scratch. Of&nbsp;
course, these tools can be used together.&nbsp;&nbsp;
For example, Terraform, a provisioning tool,&nbsp;
and Ansible, a configuration management tool,&nbsp;&nbsp;
often work as a pair. Terraform&nbsp;
provisions infrastructure, creating VMs,&nbsp;&nbsp;
network infrastructure, storage, and other&nbsp;
resources, while Ansible provides ongoing&nbsp;&nbsp;
configuration and management, maintaining&nbsp;
those systems. With all of that said,&nbsp;&nbsp;
both Terraform and Ansible have some overlap. Each&nbsp;
can perform both provisioning and configuration&nbsp;&nbsp;
management to some extent. But their strengths&nbsp;

One major difference between infrastructure&nbsp;
provisioning and management tools is the&nbsp;&nbsp;
approach they take with the infrastructure:&nbsp;
is the infrastructure mutable or immutable?&nbsp;&nbsp;
Configuration management tools typically&nbsp;
use a mutable infrastructure approach. This&nbsp;&nbsp;
means that infrastructure can be modified after&nbsp;
deployment, such as applying updates, patches,&nbsp;&nbsp;
or configuration changes. In other words, changes&nbsp;
are made in place, meaning existing resources are&nbsp;&nbsp;
updated rather than replaced. This is just like&nbsp;
when you manually modify a device’s configuration&nbsp;&nbsp;
in the CLI. So, when using a mutable approach,&nbsp;
if you want to update the config of this VM,&nbsp;&nbsp;
Ansible or a similar tool would simply update the&nbsp;
config while leaving the VM itself in place. Only&nbsp;&nbsp;
the relevant settings are changed. So, what’s&nbsp;
the alternative to mutable infrastructure?&nbsp;&nbsp;
Provisioning tools like Terraform typically&nbsp;
employ an immutable infrastructure approach.&nbsp;&nbsp;
This means that infrastructure cannot be changed&nbsp;
after deployment. So, any changes you make involve&nbsp;&nbsp;
replacing the previous resource with a new one. If&nbsp;
you want to update the configuration of this VM,&nbsp;&nbsp;
Terraform or a similar tool would have to&nbsp;
create an updated instance of the VM with&nbsp;&nbsp;
the configuration changes applied, and then&nbsp;
destroy the old instance. One benefit of this&nbsp;&nbsp;
approach is that there is no configuration drift.&nbsp;
Each deployment starts from a fresh, predefined&nbsp;&nbsp;
state. There’s no opportunity for individual&nbsp;
changes to accumulate with each configuration.&nbsp;&nbsp;
Configuration drift isn’t a major issue when&nbsp;
Ansible or other mutable infrastructure tools&nbsp;&nbsp;
are properly managed. But Terraform’s immutable&nbsp;
approach virtually eliminates the risk altogether
The final concept we’ll cover before focusing&nbsp;
on Terraform itself is the difference between&nbsp;&nbsp;
a procedural and declarative approach. Some&nbsp;
tools take a procedural approach to managing&nbsp;&nbsp;
infrastructure, also called an imperative&nbsp;
approach. This means that the tool follows&nbsp;&nbsp;
explicit steps in a specific order to achieve the&nbsp;
desired outcome. So the user, the engineer using&nbsp;&nbsp;
the tool, must define each specific action in the&nbsp;
correct order to configure the infrastructure.&nbsp;&nbsp;
This provides greater control than the declarative&nbsp;
approach we’ll cover next, but is more demanding&nbsp;&nbsp;
on the user. They have to know exactly which steps&nbsp;
are required to achieve their desired end result,&nbsp;&nbsp;
and in which order those steps should be&nbsp;
executed. Then there is the declarative approach,&nbsp;&nbsp;
which defines the desired end state instead&nbsp;
of the specific steps. The tool itself,&nbsp;&nbsp;
such as Terraform, figures out the steps needed to&nbsp;
achieve the goal. This makes it easier to maintain&nbsp;&nbsp;
and ensure consistency across deployments because&nbsp;
the user doesn’t have to worry about ordering&nbsp;&nbsp;
individual commands and configurations on each&nbsp;
platform. They only have to think about defining&nbsp;&nbsp;
what they want the final result to be. So, to&nbsp;
summarize, A procedural approach focuses on how&nbsp;&nbsp;
to make changes with explicit steps. For example,&nbsp;
when configuring a new router, configure the&nbsp;&nbsp;
router’s hostname with HOSTNAME R1. Configure the&nbsp;
IP address of G0/1 with the IP ADDRESS command.&nbsp;&nbsp;
Enable the interface with NO SHUTDOWN, , and&nbsp;
so on. These are the specific configurations&nbsp;&nbsp;
needed on a Cisco router, and the steps would be&nbsp;
different if it was a Juniper router, for example.&nbsp;&nbsp;
A declarative approach, on the other hand, focuses&nbsp;
on what the final state should be. You can tell&nbsp;&nbsp;
Terraform “Create a router named R1 with IP&nbsp;
address 192.168.1.1/24 on its G0/1 interface,&nbsp;&nbsp;
ensuring the interface is enabled”. Terraform&nbsp;
should then figure out the necessary steps to&nbsp;&nbsp;
achieve that end state. So, I’ve mentioned&nbsp;
that Terraform uses a declarative approach,&nbsp;&nbsp;
but you’re probably wondering about the others.&nbsp;
Here’s a table. I’ve categorized Ansible and&nbsp;&nbsp;
Chef as procedural, but it’s worth noting that&nbsp;
they do have some declarative elements as well,&nbsp;&nbsp;
especially Ansible. People who use Ansible&nbsp;
a lot might get angry at me for categorizing&nbsp;&nbsp;
Ansible as procedural. But Red Hat, the&nbsp;
company that owns and develops Ansible,&nbsp;&nbsp;
also says it is procedural, so for your CCNA&nbsp;
studies I’d say it’s best to consider Ansible&nbsp;&nbsp;
a procedural tool. Meanwhile, Terraform&nbsp;
and Puppet use a declarative approach,&nbsp;&nbsp;
allowing you to define the desired end state&nbsp;
and let the tool figure out the rest for you.

As I’ve mentioned a few times, it is primarily&nbsp;&nbsp;
a provisioning tool, focused on deploying&nbsp;
infrastructure resources on various cloud and&nbsp;&nbsp;
on-prem platforms. In Terraform, these platforms&nbsp;
are called providers and include AWS, Azure,&nbsp;&nbsp;
GCP, Kubernetes, and many more; it supports over&nbsp;

Like Ansible, Terraform uses a push model and&nbsp;&nbsp;
is agentless. It doesn’t require a software agent&nbsp;

now let’s see how Terraform actually works. At&nbsp;
the core of Terraform is Terraform Core; this is&nbsp;&nbsp;
the main Terraform software, typically running on&nbsp;
a Linux server, that processes configurations and&nbsp;&nbsp;
interacts with providers. An engineer creates&nbsp;
Terraform configuration files, which specify&nbsp;&nbsp;
the desired end state in a declarative fashion.&nbsp;
It’s up to Terraform Core to take that desired&nbsp;&nbsp;

the current state of the infrastructure&nbsp;

actions it needs to take, and in which order,&nbsp;&nbsp;
to achieve the desired result. To summarize,&nbsp;
these are Terraform’s four key components,&nbsp;&nbsp;
all working together to provision and manage&nbsp;
infrastructure: Terraform Core, the core software&nbsp;&nbsp;
developed by HashiCorp. Configuration files,&nbsp;
which an engineer creates to define the desired&nbsp;&nbsp;
end state of the infrastructure. The state&nbsp;
file, which tracks the current state of the&nbsp;&nbsp;
deployed infrastructure. And the providers, the&nbsp;
platforms on which the infrastructure is deployed.
Let’s cover a few more key points about Terraform.&nbsp;
The basic Terraform workflow consists of three&nbsp;&nbsp;
main steps. The first is “write”, in&nbsp;
which you define the desired end state&nbsp;&nbsp;
of your infrastructure resources in configuration&nbsp;

analyzes your configuration files and shows you&nbsp;
exactly what changes will be made. This lets&nbsp;&nbsp;
you review and confirm the changes before applying&nbsp;

executes the plan to provision and manage the&nbsp;
infrastructure resources. That’s basically how it&nbsp;&nbsp;
works. If you think back to Ansible, hopefully you&nbsp;
remember that Ansible itself is written in Python,&nbsp;&nbsp;
while its playbooks are written in YAML. So&nbsp;

an example of a domain-specific language, DSL,&nbsp;&nbsp;
which is specialized for a particular purpose,&nbsp;

video. Since DSLs are designed for specific&nbsp;
tasks, they make complex operations simpler,&nbsp;&nbsp;
and they require less effort compared to&nbsp;

Go. The downside is that HCL is another thing&nbsp;
you have to learn specifically to use Terraform,&nbsp;&nbsp;
but it’s relatively simple compared to learning&nbsp;
a full programming language. So, the key points&nbsp;&nbsp;
to remember here are the three steps of the&nbsp;
Terraform workflow: write, plan, and apply.&nbsp;&nbsp;
And also remember that Terraform Core is written&nbsp;
in Go, while the configuration files you make are&nbsp;&nbsp;
written in HashiCorp Configuration Language,&nbsp;
