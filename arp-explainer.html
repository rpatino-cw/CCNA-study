ARP Table Explainer

How a device on your LAN asks "who has this IP?" and how that honest broadcast gets weaponized.

What problem does ARP solve?

Your OS knows the destination IP address (Layer 3), but Ethernet only forwards frames by MAC address (Layer 2). ARP bridges the two: broadcast a "who has 192.168.1.30?" to the subnet, and the owner replies with its MAC. The mapping is cached for a few minutes, then re-learned. Because the protocol has no authentication, anything on the LAN can lie, which is the basis of every ARP attack.

Six-step resolution

1
Check cache. If the mapping exists and isn't expired, go straight to step 6.
2
Build request. "Who has 192.168.1.30? Tell 192.168.1.10."
3
Broadcast. Dest MAC = FF:FF:FF:FF:FF:FF. Switches flood it out every port in the VLAN, including across the trunk to the far switch. Every NIC in the broadcast domain processes it.
4
Target replies. Only the host with that IP answers, unicast to the requester.
5
Cache. Add as Dynamic. Age starts; default 4 hours on Cisco, ~2 min on Windows.
6
Send the original frame. L2 now knows where to go.
Concept

Is the switch-to-switch trunk doing ARP? No.

The trunk in the diagram only carries and floods frames. The switches are not "doing ARP," they are forwarding the ARP frames the hosts generate. ARP is a host and router function: it resolves an IP to a MAC (Layer 3 to Layer 2). Only end devices and routers build an ARP table.

Device / roleDoes ARP?What it actually usesNote
Host / RouterYesARP cache (IP-to-MAC)End devices and routers resolve an IP to a MAC before sending. A router does build an ARP table.
Layer 2 switch (forwarding)NoMAC address table / CAM (MAC-to-port)Built by reading source MACs, not by doing ARP. A switch never needs an ARP entry to forward a frame.
802.1Q trunkNoVLAN tag on each frameJust tags frames with a VLAN ID so multiple VLANs share one link. It floods the ARP broadcast to the far switch because both switches are in the same VLAN, one broadcast domain.
Switch SVI / mgmt IPYes, for itself onlyARP for its own management trafficThe SVI (VLAN interface used for SSH/Telnet) makes the switch act as a host for its own management IP, not for forwarding decisions.

This is the exam trap: do not confuse the ARP cache (host or router, IP-to-MAC) with the switch MAC / CAM table (switch, MAC-to-port). A Layer 2 switch does not need an ARP entry to forward.

How a switch learns MACs (the CAM table)

No request is sent. The switch reads the source MAC of every frame and records MAC to port. If the destination MAC is unknown it floods out all other ports (unknown unicast); once learned, it forwards out one port only. What the switch does

1
Read the source MAC. Learn sender MAC to ingress port. Add a Dynamic CAM entry. No request, just listening.
2
Look up the dest MAC. Is it already in the CAM table?
3
Unknown, so flood. No entry: send out every port except the one it arrived on (unknown unicast flooding).
4
Known, so forward. Entry exists: send out that one port only. This is never ARP.

Security

ARP attacks & mitigations

ARP has zero authentication. On a shared segment, a malicious host can impersonate anyone. The fixes live at the switch, not the protocol.

AttackHow it worksImpactMitigation
ARP SpoofingAttacker sends unsolicited replies claiming to own a victim's IP.MITM, attacker sees and modifies traffic destined for the impersonated host.Dynamic ARP Inspection (DAI) + DHCP Snooping on Cisco switches. Static ARP for routers/servers.
ARP PoisoningPersistent spoofing, attacker keeps refreshing the fake entry so caches never recover.Long-lived MITM, credential sniffing, session hijacking.DAI + arpwatch to alert on mapping changes. TLS at the app layer neutralizes most sniffing.
ARP FloodingFlood switch CAM table with bogus entries until legit ones are evicted.Switch fails open → forwards all frames as a hub. Attacker captures everything.Port security (max MACs per port), storm control, 802.1X.
Gratuitous ARP abuseAttacker sends gratuitous ARPs claiming the gateway's IP.All hosts on VLAN route through the attacker on the next refresh.DAI, private VLANs, host-based ARP monitoring.

Exam traps to memorize

ARP is a broadcast, so it stays inside the local subnet and VLAN: it never crosses a router. For a remote destination the host does not ARP for the far host, it ARPs for the default gateway's MAC, then sends the frame to the router while keeping the remote IP. Do not confuse the ARP cache with the switch MAC address table. A switch does not use ARP to forward; it forwards on its MAC (CAM) table, which it builds by reading source MACs. ARP is a host and router function (IP-to-MAC), the CAM table is the switch function (MAC-to-port). A router does build an ARP table; a Layer 2 switch does not need one for forwarding.

Remember the asymmetry: the ARP request is a broadcast (dest MAC all Fs), but the reply is a unicast straight back to the requester. And in IPv6 there is no ARP at all, NDP (ICMPv6 neighbor solicitation and advertisement over multicast) replaces it.

Watch out

Common mistakes

✗"ARP works across subnets."

Reality: ARP is link-local. Across subnets, the sender ARPs for its default gateway, and the router ARPs for the destination on the far side.

✗"Gratuitous ARP is always malicious."

Reality: Legitimate uses include IP conflict detection on boot and HA failover (VRRP/HSRP virtual IP migration). It's only suspicious when unsolicited and mismatched.

✗"HTTPS protects me from ARP poisoning."

Reality: TLS protects payload confidentiality and identity. But the attacker still controls which packets you see, they can drop, reorder, or proxy-downgrade. Certificate pinning and HSTS close most gaps.

✗"Static ARP stops all spoofing."

Reality: Only for that one entry. Attackers can still target other hosts, flood the switch, or target the gateway's own cache. DAI is the systemic fix.

✗"ARP packets are IP packets."

Reality: ARP is its own Layer-2 protocol (EtherType 0x0806), not encapsulated in IP. That's why it works before IP connectivity is established.

✗"ARP cache ages out instantly."

Reality: Timers vary wildly: Windows ~2 min idle, Linux ~60 s (base_reachable_time), Cisco IOS 4 hours by default. Cisco's long timer is a common source of stale-entry confusion during HA events.
