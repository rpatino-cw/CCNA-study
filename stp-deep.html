STP Deep Dive

Why a flat Layer-2 network needs a referee — and exactly how 802.1D, 802.1w, and Cisco's flavors do that job.

What problem does STP solve?

Ethernet frames have no TTL. A loop in a Layer-2 topology turns one broadcast into a storm that saturates the link, destabilizes the MAC address table, and duplicates unicast frames forever — all in milliseconds. STP runs a distributed algorithm across every switch to elect one root, calculate the lowest-cost path back to it from every other switch, and logically block the redundant ports so the topology becomes a tree. When a link fails, STP recomputes and unblocks a backup port. RSTP (802.1w) and Cisco's Rapid-PVST+ do the same job in seconds instead of ~50 s.

01 · Why STP exists

The three loop disasters

L2 has no TTL and no IP-style routing decision. Drop a single redundant link in without STP and you get all three failures simultaneously — usually within seconds.

1. Broadcast storm

One broadcast (ARP, DHCP, CDP) loops endlessly through the redundant path. Each switch re-floods it on every other port. CPU and links saturate. Sometimes hits 100% utilization in under a second.

2. MAC table instability

Same source MAC arrives on two different ports as the broadcast loops. The switch flips the MAC entry back and forth between ports — dynamic learning collapses. Forwarding decisions become random.

3. Duplicate frame delivery

Unicast frames also loop. The destination receives multiple copies of every frame. Upper-layer apps see corruption or session resets. TCP handles it; UDP voice/video does not.

Why a router is fine

Routers decrement the IP TTL on every hop and drop at zero. They also don't flood broadcasts past the L3 boundary. A loop at L3 burns out in 64–255 hops. A loop at L2 burns until you unplug a cable.

Mental model

STP is not "off" by default — every Cisco switch ships with PVST+ or Rapid-PVST+ enabled on every VLAN. You only meet STP when you misconfigure it (wrong root, wrong cost, PortFast on an uplink) or when it converges too slowly for an outage.

02 · BPDU

Bridge Protocol Data Unit

Switches don't share routing tables — they share BPDUs. Every switch periodically multicasts its view of "who is root, what's my cost to root, what's my BID, what port is this." Compare incoming BPDUs to your own and the topology emerges.

Two BPDU types

  • Configuration BPDU — sent every 2 s (Hello timer) by the root. Carries root BID, root path cost, sender BID, port ID, message age, max age, hello, forward delay. This is the "I'm root, here's my view" advertisement that propagates the tree.
  • TCN BPDU (Topology Change Notification) — tiny BPDU sent upstream toward the root when a port changes state. Triggers a MAC table flush. RSTP replaces this with a TC flag inside the regular BPDU.

Destination MAC: 01:80:C2:00:00:00 — the IEEE bridge group multicast. Switches consume it; they do not forward it as a normal frame.

Gotcha — only the root sets timers

Hello/MaxAge/ForwardDelay come from the root's BPDU, not from the local switch's config. Setting spanning-tree vlan 10 hello-time 1 on a leaf switch does nothing unless that switch becomes root. To change tree-wide timers, set them on the root or use root primary + timer commands.

03 · Bridge ID

Bridge ID — the identity used in every comparison

Every BID is 8 bytes. Originally 2 B priority + 6 B MAC. PVST+ split the priority into 4 b priority + 12 b VLAN ID so one chassis can advertise a different BID per VLAN.

Original 802.1D BID (8 bytes) ┌────────────────┬────────────────────────────────────────────┐ │ Priority (2 B) │ MAC Address (6 B) │ │ 0–65535 │ Burned-in chassis MAC │ └────────────────┴────────────────────────────────────────────┘ PVST+ extended BID (Cisco default — 802.1t) ┌────────┬────────────┬────────────────────────────────────────┐ │ Pri │ Sys-ID-Ext │ MAC Address (6 B) │ │ 4 bits │ 12 bits │ │ │ × 4096 │ = VLAN ID │ │ └────────┴────────────┴────────────────────────────────────────┘ Effective priority field = (Pri × 4096) + VLAN ID Default Pri = 32768 → VLAN 1 BID priority = 32769 → VLAN 10 BID priority = 32778

Why priority is restricted to multiples of 4096

The low 12 bits hold the VLAN. Only the top 4 bits are the configurable priority. Valid priority values are 0, 4096, 8192, 12288 … 61440. Default is 32768. IOS rejects anything not a multiple of 4096.

spanning-tree vlan 10 priority 4096 sets BID priority for VLAN 10 to 4096 + 10 = 4106. The "+ 10" is the sys-id-ext baked into the BID; you cannot remove it on a Cisco PVST+ switch.

Lower BID wins — every comparison, no exceptions

  1. Compare priority first.
  2. If tied, compare MAC address (lower wins).

This single rule decides root election, tiebreakers in port-role selection, and even who becomes designated on a shared segment.

04 · Election

Root-bridge election walkthrough

Every switch boots assuming it is the root. It sends BPDUs claiming root. Then it listens. As soon as it hears a better BPDU (lower BID), it stops claiming root and starts relaying that better BPDU. After a few hello cycles, the entire L2 domain agrees on one root.

1
Boot. Switch sends BPDU with Root BID = own BID, Root Path Cost = 0.
2
Receive a superior BPDU — one with a lower Root BID. Stop claiming. Cache that root.
3
Forward it. On every other port, send BPDUs with the cached root BID and the incremented path cost = received cost + ingress port cost.
4
Pick the root port (RP) = the port on which the lowest-cost path to root arrived. Tiebreakers (in order): lowest sender BID → lowest sender port priority → lowest sender port number.
5
Pick designated ports (DP) per segment. On each link, the side with the better BPDU advertisement is designated. Root's ports are always DP.
6
Block leftovers. Any port that is neither RP nor DP becomes Blocking (802.1D) or Alternate/Backup (802.1w). It still hears BPDUs but never forwards data.
Forcing the right root

Never trust the default 32768 priority. The oldest switch with the lowest MAC wins by accident — usually the wrong box. Hard-pin the core: spanning-tree vlan 1-1005 priority 4096 on the intended root, 8192 on the secondary.

Worked example — three-switch triangle

Priorities (default 32768) + last MAC byte: SW1: 32768.AA SW2: 32768.BB SW3: 32768.CC SW1 ──── SW2 \ / \ / SW3 Step 1: All three claim root → SW1 wins (lowest MAC AA < BB < CC). Step 2: SW2 root port = port toward SW1, cost 4 (1 Gb). SW3 root port = port toward SW1, cost 4 (1 Gb). Step 3: SW1↔SW2 link: SW1 designated. SW2 side: root port (already chosen). SW1↔SW3 link: SW1 designated. SW3 side: root port. SW2↔SW3 link: tie at cost 4 from each side → SW2 wins (lower MAC). SW2 = designated, SW3 = blocking (alternate). Result: One blocked port on SW3, toward SW2. Loop broken.

Gotcha — root path cost is added on ingress

BPDU arrives saying "cost 19." The receiving switch adds the cost of the port it arrived on, not the egress port. If a 1-Gb port receives a BPDU from a 100-Mb upstream link, the local cost increment is the 4 for the local 1-Gb port, not 19. New total = received 19 + local 4 = 23.

06 · 802.1D states

Port states & the 50-second convergence

Classic 802.1D walks every port through five states. Total time for a port to go from "link up" to "forwarding data" is up to 50 seconds: 20 s Max Age (if waiting on a stale root) + 15 s Listening + 15 s Learning.

The 50-second math (worst case after a link failure)

  • 20 s — Max Age. Indirect failure. The blocked port stops hearing BPDUs from root and waits Max Age before declaring the path lost.
  • 15 s — Listening. Sends/receives BPDUs to compute new roles.
  • 15 s — Learning. Populates MAC table without forwarding to avoid floods.

Direct failure (port goes down on the local switch) skips Max Age — convergence drops to ~30 s. Either way, RSTP fixes this.

07 · Topology change

Topology Change Notification & MAC flush

Why MACs must flush

Suppose a host's traffic was egressing via SW3's blocking port becoming forwarding. Every switch in the path has stale MAC entries pointing at the old route. Without flushing, frames go to a black hole until the default 5-minute aging timer expires.

802.1D TCN flow

  1. Switch detects a port going to/from forwarding.
  2. Sends a TCN BPDU upstream toward the root via its root port.
  3. Each upstream switch acknowledges with TCA flag and forwards the TCN further upstream.
  4. Root receives the TCN. For the next Forward Delay + Max Age seconds (default 35 s), root sets the TC flag in every BPDU it sends.
  5. Every switch receiving a BPDU with TC set shortens its MAC aging timer from 300 s to 15 s. Stale entries die quickly; new entries are learned at the new path.
Gotcha — flapping access ports flood the tree

An access port going up/down emits a TCN every time. Multiply by hundreds of users plugging laptops in — the tree triggers a domain-wide MAC age-shortening on every plug event. PortFast suppresses TCNs for edge ports.

08 · RSTP 802.1w

Rapid STP — same tree, three states, ~6 s convergence

RSTP is wire-compatible with 802.1D — it interoperates by falling back. The differences are in how the tree is built and how fast it heals.

Discarding
stable
Combines 802.1D Disabled + Blocking + Listening. Drops data, no MAC learning, can hear BPDUs.
Learning
transient
Builds MAC table. Same as 802.1D.
Forwarding
stable
Same as 802.1D.

Why it's fast — proposal/agreement handshake

Instead of waiting timers, two RSTP switches connected by a point-to-point full-duplex link negotiate directly:

  1. Upstream switch sends a BPDU with the Proposal flag set, claiming designated.
  2. Downstream switch syncs — temporarily blocks every other non-edge port to guarantee no loop forms.
  3. Downstream replies with Agreement flag set on its new root port.
  4. Upstream port goes straight to forwarding. The wave repeats outward.

Convergence under failure: typically 1–6 seconds, often sub-second with healthy peers.

Three RSTP link types

Link typeDetected byBehavior
Point-to-PointFull duplexEligible for fast proposal/agreement.
SharedHalf duplex (hub)Falls back to 802.1D timers — no fast convergence.
EdgeManual (PortFast)Skips listening/learning. Goes straight to forwarding. No TCN generated.

RSTP topology changes

RSTP doesn't use TCN BPDUs. When a non-edge port transitions to forwarding, the originating switch:

  • Sets the TC flag directly in regular BPDUs for 2× Hello seconds.
  • Floods these BPDUs out every non-edge designated port and the root port (not just toward root).
  • Every switch receiving a TC-flagged BPDU flushes all MACs on every port except the one that received the BPDU.

Faster convergence + faster MAC re-learning = the new path actually carries traffic in a couple of seconds.

Backwards compatibility

If an RSTP switch detects 802.1D BPDUs on a port, it falls back to legacy timers on that port only. The rest of the tree stays RSTP. Mix carefully — one legacy box can drag a region back to 50-second land.

09 · Cisco flavors

PVST+ vs Rapid-PVST+ vs MSTP

Cisco runs one STP instance per VLAN. Up to 1005 VLANs → up to 1005 trees, each with its own root, its own port states, its own BPDUs. Great for traffic engineering, brutal on CPU at scale. MSTP fixes the scale problem.

PVST+ (legacy default)

  • One STP per VLAN.
  • 802.1D timers — convergence ~50 s.
  • Cisco proprietary BPDU encoding.
  • BID uses sys-id-ext (priority + VLAN).
  • Default on older IOS.

Rapid-PVST+ (Cisco default today)

  • One STP per VLAN, but uses 802.1w mechanics.
  • Convergence ~1–6 s.
  • Three port states (Discarding / Learning / Forwarding).
  • Backwards compatible with 802.1D.
  • Enable: spanning-tree mode rapid-pvst.

MSTP / 802.1s

  • Maps many VLANs into a few instances.
  • One STP run per instance — CPU-friendly at scale.
  • Standards-based, multi-vendor.
  • Same RSTP fast convergence underneath.
  • Enable: spanning-tree mode mst + region config.
Per-VLAN load balancing

Make SW1 root for odd VLANs and SW2 root for even VLANs. Each VLAN's blocked port is on a different switch, so both uplinks carry traffic. spanning-tree vlan 10,20,30 priority 4096 on SW1 + spanning-tree vlan 40,50,60 priority 4096 on SW2.

Never PortFast a switch-to-switch link

PortFast assumes there is no switch on the other end. If you enable it on an inter-switch port, BPDUs are missed during the fast transition window and a loop can form. BPDU Guard is the safety net — pair them globally with spanning-tree portfast bpduguard default.

Root Guard vs BPDU Guard

BPDU Guard assumes no BPDU at all. Edge ports.

Root Guard assumes BPDUs are fine, but never a superior one. Trunk-toward-access ports where you don't trust the downstream not to claim root.

Loop Guard vs UDLD

Loop Guard reacts when BPDUs stop. Software-only — works on any link.

UDLD actively probes the fiber pair and detects unidirectional failure even when BPDUs aren't expected. Layer-2 protocol on fiber.

Best practice on fiber inter-switch links: both.

12 · Exam & study

Common exam traps

Six gotchas worth a question each

  • Priority isn't 32768 — it's 32768 + VLAN ID in any output. Memorize 32769, 32778, 32788…
  • Root path cost is added on ingress. Receiving cost 19 + my port cost 4 = 23, regardless of upstream interface.
  • Cost values: 100 Mbps = 19, 1 Gbps = 4, 10 Gbps = 2. Long method is different — match the question's context.
  • Tiebreaker order for choosing root port when costs tie: lower sender BID → lower sender port priority → lower sender port number.
  • BPDU Guard ⇒ err-disable. Root Guard ⇒ root-inconsistent. Loop Guard ⇒ loop-inconsistent. Don't swap them on a multiple-choice.
  • Rapid-PVST+ port states are 3, not 5. Discarding / Learning / Forwarding. The exam loves mixing 802.1D and 802.1w terminology.

One-page study card

  • BPDU dest MAC: 01:80:C2:00:00:00
  • Hello / MaxAge / ForwardDelay: 2 / 20 / 15 seconds (root sets them).
  • 802.1D convergence: ~50 s. RSTP: ~1–6 s.
  • Bridge ID: 4 b priority + 12 b VLAN ID + 6 B MAC. Default priority 32768, multiples of 4096.
  • Lower BID wins. Tie on priority → lower MAC wins.
  • Port roles: Root, Designated, Alternate (RSTP) / Blocking (legacy), Backup, Disabled.
  • States 802.1D: Disabled → Blocking → Listening → Learning → Forwarding.
  • States RSTP: Discarding → Learning → Forwarding.
  • PortFast: edge only. Skips listening/learning.
  • BPDU Guard: err-disables a PortFast port that hears a BPDU.
  • Root Guard: blocks superior BPDU. Recovers on its own.
  • Loop Guard: protects against silent failure (BPDUs stop arriving).
  • Cisco default mode: Rapid-PVST+ on modern IOS. One instance per VLAN.
  • Force a root: spanning-tree vlan X priority 4096.
