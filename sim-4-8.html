SSH Setup vs Telnet Sim

Telnet was fine in 1973 — when the only people on the wire were trusted. Today, every byte you type into Telnet is plaintext on every router between you and the device, including your password. SSH wraps the same shell in a Diffie-Hellman key exchange and a symmetric cipher. Same prompt, but eavesdroppers see encrypted noise.

SSH Setup Sequence

1. TCP 3-way handshake client SYN → server SYN+ACK → client ACK on port 22
2. Version exchange "SSH-2.0-OpenSSH_9.0" advertised both ways
3. Algorithm negotiation kex=curve25519, cipher=aes256-gcm, mac=hmac-sha2-256, host-key=rsa
4. Key exchange Diffie-Hellman — both sides derive a shared secret without sending it
5. Host key auth server signs the DH hash with its RSA key. Client checks fingerprint against ~/.ssh/known_hosts
6. User auth password OR public key (id_rsa.pub already in authorized_keys)
7. Encrypted channel all subsequent traffic AES-encrypted with the DH-derived session key

Cisco Config

R1(config)# hostname R1 R1(config)# ip domain-name lab.local R1(config)# crypto key generate rsa modulus 2048 R1(config)# username admin privilege 15 secret StrongPass! R1(config)# ip ssh version 2 R1(config)# ip ssh time-out 60 R1(config)# ip ssh authentication-retries 3 R1(config)# line vty 0 4 R1(config-line)# transport input ssh R1(config-line)# login local R1(config-line)# exec-timeout 5 0 R1(config-line)# end ! Why each command: ! hostname + domain-name → required for RSA key generation ! crypto key generate rsa modulus 2048 → real entropy, ≥1024 for SSHv2 ! ip ssh version 2 → reject SSHv1 (vulnerable) ! transport input ssh → DISABLES Telnet on the vty lines ! login local → use local username/password DB
