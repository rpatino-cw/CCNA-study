CCNA 6.6 · Spotlight

Ansible — the agentless one

The CCNA exam asks you to recognize the capabilities of Puppet, Chef, and Ansible. Most candidates lose this question because the three tools blur together. This guide makes Ansible stick — what it is, how it ships configs to a Cisco router, and how it differs from the other two — with a real playbook you'd run on test gear.

Exam objective 6.6

Recognize the capabilities of configuration management mechanisms Puppet, Chef, and Ansible

You will not be asked to write a playbook. You will be asked: agent vs agentless, push vs pull, default port, language. Memorize that row of the table at the bottom — that's the test.

Push vs Pull — the diagram that earns the point

Ansible pushes. Puppet and Chef pull. That single sentence is worth a question on test day.

Two files, one config push

Every Ansible run reads two pieces — an inventory (who) and a playbook (what). The example below adds VLAN 10 to a Cisco switch.

1 · inventory.ini — who
# groups of devices, plain text
[switches]
sw1 ansible_host=10.0.1.2
sw2 ansible_host=10.0.1.3

[switches:vars]
ansible_user=admin
ansible_network_os=ios
ansible_connection=network_cli
2 · vlan.yml — what
---
- name: Add VLAN 10 to switches
  hosts: switches
  gather_facts: no

  tasks:
    - name: Create VLAN 10
      cisco.ios.ios_vlans:
        config:
          - vlan_id: 10
            name: USERS
            state: active
        state: merged
3 · run it
# one command — control node pushes to every switch in the group
$ ansible-playbook -i inventory.ini vlan.yml

# expected output
PLAY [Add VLAN 10 to switches] ********************************
TASK [Create VLAN 10] *****************************************
changed: [sw1]
changed: [sw2]

PLAY RECAP ****************************************************
sw1   : ok=1  changed=1  failed=0
sw2   : ok=1  changed=1  failed=0
Why this is safe to re-run

Idempotency — the word that wins the question

Run the playbook ten times in a row. VLAN 10 still exists exactly once. Ansible checks the current state before changing anything — if it already matches, the task reports ok instead of changed. That property is called idempotency and it's true for all three tools (Ansible, Puppet, Chef).

Ad-hoc — one-liners without a playbook

Sometimes you just need to ping every device. Ansible has a one-line mode for that.

# are all switches reachable?
$ ansible switches -i inventory.ini -m ping

# run a Cisco "show" command on every switch
$ ansible switches -i inventory.ini -m cisco.ios.ios_command \
    -a "commands='show ip interface brief'"

What happens when you hit enter

  1. Ansible reads inventory.ini and resolves the switches group to two IPs.
  2. Ansible reads vlan.yml and parses the YAML into Python tasks.
  3. Control node opens an SSH session to sw1 on port 22 — no agent answers, just IOS.
  4. Ansible runs show vlan brief first to see what's already there. This is the idempotency check.
  5. VLAN 10 missing → Ansible sends vlan 10 / name USERS to the running config.
  6. Reports changed back to the control node, closes the SSH session, repeats for sw2.
  7. Final play recap prints. No daemon left running on either switch.

One-line summary

Memorize this exact sentence

Ansible is agentless, push-based, uses YAML over SSH, and needs no master server. If you can recite that, you've got the question.
