3.2 Forwarding Decision — Read This Before Anything Else

Cisco 200-301 §3.2.a-c. Three rules. One order. Memorize cold: 1) Longest prefix match → 2) Lowest AD → 3) Lowest metric. Read top-to-bottom, then drill until 95%+.

1. The 30-Second Answer read this twice

A router's forwarding decision answers ONE question: "Of all routes that match the destination IP, which goes in the routing table?" Three rules, in order: 1) Longest prefix match wins. 2) If tied on prefix, lowest AD wins. 3) If tied on AD (within one protocol), lowest metric wins.
  • LPM (3.2.a) — match the destination against every route. The route with the MOST mask bits set wins. Always. First filter.
  • AD (3.2.b) — only matters when TWO protocols learn the SAME prefix. Lower AD = more trusted. Static (1) beats RIP (120). Static beats OSPF (110). Connected (0) beats everything.
  • Metric (3.2.c) — only matters WITHIN one protocol. OSPF cost 65 vs OSPF cost 100 → cost 65 wins. Cross-protocol metric comparisons are MEANINGLESS.

Memorize: LPM → AD → METRIC — never swap the order.

2. Decision Pipeline at a Glance visual

The 3 rules run TOP-DOWN — LPM filters first; only routes for the same longest match are compared on AD; only routes from the same protocol with same AD are compared on metric. Skip a step → wrong answer.

3. Longest Prefix Match — 3.2.a most-tested

"Longest prefix match" = match destination IP against every route's network/mask. The route with the MOST mask bits set (longest prefix) wins. More specific = better.

Routing table: 10.0.0.0/8 via 10.0.0.1 10.10.0.0/16 via 10.0.0.2 10.10.20.0/24 via 10.0.0.3 10.10.20.50/32 via 10.0.0.4 Packet for 10.10.20.50 → which route is used? /8 matches? YES — 10 = 10 /16 matches? YES — 10.10 = 10.10 /24 matches? YES — 10.10.20 = 10.10.20 /32 matches? YES — 10.10.20.50 = 10.10.20.50 /32 = longest prefix → WINS. Forwarded via 10.0.0.4.
  • Always check LPM first. Even if a /8 has lower AD or better metric, a matching /24 still wins because it's more specific.
  • Default route (0.0.0.0/0) = the SHORTEST possible prefix. Last resort. Used only when no other route matches.
  • Host route (/32) = the LONGEST possible prefix. Beats everything else for that one IP.
  • LPM happens at data-plane lookup time, every single packet. The router doesn't reconsider AD/metric per-packet — those were already used to decide what got into the FIB.
Trap to recognize
Cisco loves: "a packet for 10.10.20.50 — there's a /16 with low cost and a /24 with high cost. Which wins?" → /24 wins. Always. LPM is checked before AD or metric.

4. Administrative Distance — 3.2.b cross-protocol tiebreaker

AD only matters when TWO different protocols learn the SAME prefix. Lower AD wins. Whichever wins gets installed in the RIB; the loser is suppressed (still tracked by the protocol but not used for forwarding).

Floating static = a static route with manually-raised AD, used as backup. Example: ip route 10.0.0.0 255.0.0.0 10.10.10.99 130 — AD 130 makes it less trusted than RIP (120). Only kicks in if RIP fails. Common HA pattern.

Same AD, different protocols: impossible by default — each protocol has its own fixed AD value. AD only conflicts when manually overridden.

5. Metric — 3.2.c within-protocol tiebreaker

Metric only compares routes from the same protocol. Two OSPF routes for the same prefix → lower OSPF cost wins. Two EIGRP routes → lower composite metric wins.

ECMP (Equal-Cost Multi-Path): if two routes from the same protocol have the SAME prefix + SAME metric, BOTH go in the routing table. Traffic load-shares. Cisco default = up to 4 equal paths (configurable up to 32 via maximum-paths).

"OSPF cost 50 vs RIP hop 5 — which wins?" OSPF. AD 110 < AD 120. Metric never enters comparison across protocols.

6. RIB vs FIB — Why It Matters control vs data plane

RIB (Routing Information Base)

  • Control plane — built by routing protocols
  • Lives in CPU memory
  • Holds candidate + winning routes
  • Updated when topology changes (slow, seconds)
  • View: show ip route

FIB (Forwarding Information Base)

  • Data plane — derived FROM the RIB
  • Lives in TCAM / ASIC memory
  • Holds ONLY winning routes
  • Looked up at line rate per packet
  • View: show ip cef

CEF (Cisco Express Forwarding) = the modern lookup engine that uses the FIB. Replaces older process switching + fast switching. Default on all modern Cisco routers + L3 switches.

The forwarding decision (LPM → AD → metric) happens during RIB → FIB programming, NOT per-packet. Per-packet, the router just does TCAM lookup against the FIB. Already-decided. Wire-speed.

9. Cisco's Favorite Traps spot the wrong answer

Trap #1 — "Lowest AD always wins"
Wrong. Longest prefix wins FIRST. /24 with AD 200 still beats /16 with AD 1. AD only tiebreaks if two routes match the same longest prefix.
Trap #2 — "Lowest metric always wins"
Wrong. Metric is rule #3 — last in line. LPM first, AD second, metric third. And metric only compares within ONE protocol.
Trap #3 — "OSPF cost 5 beats RIP hop 1"
Wrong reason. OSPF wins, but because of AD, not metric. AD 110 < AD 120. Metric is irrelevant across protocols. Cross-protocol metric comparison is meaningless.
Trap #4 — "Static beats connected"
Wrong. Connected (AD 0) beats static (AD 1). Static beats EVERYTHING ELSE except connected. Backup intuition: the router can SEE the connected link, so trusts it most.
Trap #5 — "Two routes = always ECMP"
Wrong. ECMP requires same prefix + same protocol + same metric. Different protocols = AD picks one, no ECMP. Different metrics = lower wins, no ECMP. Strict rules.
Trap #6 — "Floating static is a /32 route"
Wrong. Floating static = static with raised AD. Used as backup. Configured: ip route 10.0.0.0 255.0.0.0 X.X.X.X 130 (AD 130). Not active unless primary protocol fails.
Trap #7 — "FIB and RIB are the same"
Wrong. RIB = control plane (all candidates), FIB = data plane (winners only, in hardware). Different memory regions. Different roles. Confusing them = wrong answer on architecture Qs.

10. Top 10 Must-Know if-only-this

Memorize until automatic
  1. Forwarding decision order: LPM → AD → METRIC. Never swap. Never skip.
  2. LPM = always applies. Most-specific route wins regardless of AD or metric.
  3. AD = cross-protocol tiebreaker. Only when 2+ protocols learn the same prefix.
  4. Metric = within-protocol tiebreaker. Cross-protocol metric comparisons are meaningless.
  5. AD ladder: 0/1/20/90/110/115/120/170/200 (Connected/Static/eBGP/EIGRP/OSPF/ISIS/RIP/EIGRP-ext/iBGP).
  6. Default route = 0.0.0.0/0 = shortest prefix possible. Used only when nothing else matches.
  7. ECMP rule: same prefix + same protocol + same metric = all paths installed, traffic load-shared.
  8. Floating static = static with raised AD for backup. Doesn't activate until primary route fails.
  9. RIB vs FIB: RIB = candidates (CPU/RAM, slow). FIB = winners (TCAM, line-rate). Decision runs RIB→FIB.
  10. Verification commands: show ip route X.X.X.X (control plane) + show ip cef X.X.X.X (data plane). Use both to debug forwarding.

11. Self-Check no peeking

Q1. Routing table has 10.0.0.0/8 [110/5] and 10.10.20.0/24 [120/100]. Packet destined for 10.10.20.50. Which route is used?

Reveal

10.10.20.0/24. LPM wins first. /24 is more specific than /8. AD and metric are NOT compared because LPM already decided. Refer back: §3, §9 trap #1.

Q2. Two protocols both have a route to 10.10.0.0/16. OSPF cost 50, EIGRP composite 200000. Which is installed?

Reveal

EIGRP. Same prefix → AD compared. EIGRP AD 90 < OSPF AD 110. Metric never enters across-protocol comparison. Refer back: §4, §9 trap #3.

Q3. OSPF has two paths to 10.10.0.0/16 — cost 65 and cost 100. What's installed?

Reveal

Only cost 65. Within OSPF, lower metric wins. Cost 100 sits in OSPF database but not RIB. Refer back: §5.

Q4. EIGRP has two equal-cost paths (same metric) to 10.10.20.0/24. What gets installed?

Reveal

Both — ECMP. Same prefix + same protocol + same metric = both go in. Traffic load-shares. Refer back: §5, §9 trap #5.

Q5. Where is the forwarding decision actually executed for each packet?

Reveal

FIB lookup in TCAM (CEF). The 3-rule decision (LPM/AD/metric) ran during RIB → FIB programming. Per-packet, just a fast TCAM lookup. Refer back: §6.
