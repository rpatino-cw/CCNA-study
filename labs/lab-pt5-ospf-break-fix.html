Week 5

Break and Fix OSPF

~60 min Topic 3.4 Packet Tracer · Troubleshooting
Objectives
  • 01Start from a working 3-router OSPF topology (same as Lab PT-4)
  • 02Inject each of the 5 faults listed below, one at a time, using the "break" commands
  • 03Use only show commands to diagnose the symptom before looking at the fix
  • 04Apply the fix command and confirm the adjacency returns to FULL state
  • 05After all 5 faults, restore a clean working config and verify all routes are present
Starting Topology
  Same 3-router triangle from Lab PT-4:

    R1 (1.1.1.1) ──10.12.0.0/30── R2 (2.2.2.2)
       │                                │
  10.13.0.0/30                    10.23.0.0/30
       │                                │
       └─────────── R3 (3.3.3.3) ───────┘

  All in OSPF Area 0, process 1. All adjacencies FULL before injecting faults.

Complete Lab PT-4 first, or use the same config. Before each fault: confirm show ip ospf neighbor shows all neighbors as FULL. Inject one fault at a time — fix and verify before moving to the next.

The 5 Faults
FAULT 1 Area Mismatch
Break command
R2
R2(config)# router ospf 1
R2(config-router)# no network 10.12.0.0 0.0.0.3 area 0
R2(config-router)# network 10.12.0.0 0.0.0.3 area 1
Symptom to look for
R1's neighbor table shows R2 stuck in INIT or 2-WAY — never reaches FULL. R1 still has routes through R3 but nothing learned from R2.
Diagnose with
show ip ospf neighbor — State column shows INIT or 2WAY

debug ip ospf adj — Look for "Area mismatch" in output (disable after: undebug all)
R2 — Fix
R2(config)# router ospf 1
R2(config-router)# no network 10.12.0.0 0.0.0.3 area 1
R2(config-router)# network 10.12.0.0 0.0.0.3 area 0
Both sides of a link must be in the same area. Area 0 is the backbone — single-area OSPF always uses area 0.
FAULT 2 Wrong Wildcard Mask
Break command
R1
R1(config)# router ospf 1
R1(config-router)# no network 10.13.0.0 0.0.0.3 area 0
R1(config-router)# network 10.13.0.0 0.0.0.255 area 0
Symptom
R1–R3 adjacency drops. The wildcard 0.0.0.255 matches a whole /24 — it activates OSPF on the wrong interfaces, or not on Fa0/1 at all depending on the address range.
Diagnose with
show ip ospf interface brief — Fa0/1 may not appear in OSPF

show running-config | section ospf — spot the wrong wildcard
R1 — Fix
R1(config)# router ospf 1
R1(config-router)# no network 10.13.0.0 0.0.0.255 area 0
R1(config-router)# network 10.13.0.0 0.0.0.3 area 0
/30 mask = 255.255.255.252. Inverse = 0.0.0.3. The wildcard tells OSPF which bits must match exactly.
FAULT 3 Hello/Dead Timer Mismatch
Break command
R2 — Fa0/0 (R1 link)
R2(config)# interface fastethernet 0/0
R2(config-if)# ip ospf hello-interval 5
R2(config-if)# ip ospf dead-interval 20
Symptom
R1–R2 adjacency drops and keeps flapping. OSPF requires both ends to have matching Hello (default 10s) and Dead (default 40s) timers.
Diagnose with
show ip ospf interface fa0/0 — shows Timer intervals

Compare R1's timers to R2's timers on the same link. Mismatch = adjacency fails.
R2 — Fix
R2(config)# interface fastethernet 0/0
R2(config-if)# no ip ospf hello-interval
R2(config-if)# no ip ospf dead-interval
Removing the override restores defaults (Hello=10, Dead=40). Alternatively, set matching values on both sides. Timer mismatch is one of the most common OSPF issues in the real world.
FAULT 4 Passive Interface on Transit Link
Break command
R3
R3(config)# router ospf 1
R3(config-router)# passive-interface fastethernet 0/0
Symptom
R1–R3 adjacency drops immediately. R3 stops sending Hellos on Fa0/0. The link is still "up/up" in the interface table, which makes this tricky to spot.
Diagnose with
show ip ospf interface fa0/0 — shows "Passive interface"

Interface status is up, IP is correct, but OSPF is not sending Hellos. The line "No Hellos (Passive interface)" is the giveaway.
R3 — Fix
R3(config)# router ospf 1
R3(config-router)# no passive-interface fastethernet 0/0
passive-interface is correct on loopbacks and user-facing LANs, but never on a link between two OSPF routers. The confusion between "where to use passive" is a guaranteed exam question.
FAULT 5 MTU Mismatch
Break command
R2 — Fa0/1 (R3 link)
R2(config)# interface fastethernet 0/1
R2(config-if)# ip mtu 1200
Symptom
R2–R3 adjacency gets stuck at EXSTART or EXCHANGE — neighbors never reach FULL. This is the subtlest fault because Hellos still work (they're small), but the DBD exchange fails.
Diagnose with
show ip ospf neighbor — stuck at EXSTART/EXCHANGE

show interfaces fa0/1 | include MTU — check MTU

Compare both sides. Default is 1500. One side at 1200 = mismatch.
R2 — Fix
R2(config)# interface fastethernet 0/1
R2(config-if)# no ip mtu
OSPF uses MTU during DBD (Database Description) exchange to determine max packet size. A mismatch means the larger-MTU side can't read the other's DBD packets, stalling the exchange permanently at EXSTART. In real networks this happens between Ethernet (1500) and tunnels (1400–1476).
