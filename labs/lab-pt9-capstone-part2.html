Layer 2 Requirements

STP (Rapid PVST+)

  • CORE1 root primary for VLANs 10, 30, 99 (priority 24576)
  • CORE2 root primary for VLAN 20; root secondary for others
  • Access ports: portfast + bpduguard enable
  • Distro downlinks: spanning-tree guard root

Trunking

  • All switch-to-switch: mode trunk, encapsulation dot1q
  • Native VLAN = 999 everywhere
  • DTP off: switchport nonegotiate
  • Explicit allowed-list — prune the rest

EtherChannel (LACP)

  • Po1–Po6: distro-to-access bundles (2 members each)
  • Po10: CORE1↔CORE2 L3 port-channel, no switchport
  • Mode active both sides — no static on

Port-security (access ports)

  • maximum 2 (PC + daisy-chained phone)
  • violation restrict
  • mac-address sticky
  • aging time 60 type inactivity

DHCP snooping

  • Enable globally + per VLAN (10, 20, 30, 40, 60)
  • Trust every uplink — most common outage cause
  • Access ports untrusted by default
Layer 3 / Routing

HSRPv1 (HQ user VLANs)

  • VLAN 10: CORE1 active (priority 110 preempt)
  • VLAN 20: CORE2 active (load-balance)
  • VLAN 30: CORE1 active
  • preempt delay minimum 30 — prevents boot-time thrash
  • Virtual IP = first usable of each subnet

OSPFv2 — area 0 only

  • Process 1, router-id = Loopback0 IP
  • Enable on WAN /30s + core↔distro L3 + Loopback0
  • passive-interface default + selective un-passive
  • reference-bandwidth 1000
  • MD5 auth key per interface

Summarization

  • Each site advertises ONLY its /20 summary
  • Use distribute-list out + prefix-list on WAN router
  • HQ-RTR: default-information originate always

Default route

  • HQ-RTR: ip route 0.0.0.0 0.0.0.0 203.0.113.1
  • BR/DC: learn default via OSPF
  • ISP: 203.0.113.0/30 (documentation block)
Services

DHCP

  • Server on HQ-RTR — one pool per served VLAN
  • Exclude first 10 IPs + HSRP VIP
  • Options: default-router (VIP), dns-server 8.8.8.8
  • Relay: ip helper-address <HQ-RTR> on BR/DC SVIs

NAT / PAT

  • Inside = all internal; outside = Gi0/2 (ISP)
  • ip nat inside source list NAT-INTERNAL ... overload
  • ACL must permit every site summary
  • One static NAT: DC web server → public:8080

NTP

  • HQ-RTR: ntp master 2 + MD5 key 1
  • Clients: ntp server <loopback> + same key
  • Don't forget ntp trusted-key 1

SSH + AAA

  • ip domain-name meridian.lab + RSA 2048
  • ip ssh version 2; telnet off
  • aaa new-model, login local
  • vty access-class 10 in

SNMPv3 + Syslog

  • Group MERIDIAN-RO, user monitor, SHA + AES
  • Traps to DC server
  • logging trap informational
  • logging source-interface Loopback0
ACLs

Extended: BLOCK-GUEST-INTERNAL

  • deny ip <BR-GUEST> <HQ summary>
  • deny ip <BR-GUEST> <DC summary>
  • permit ip any any (so guest → internet works)
  • Order matters — deny before permit

Standard 10: vty allowlist

  • permit HQ-MGMT subnet
  • permit BR-MGMT subnet
  • permit DC-MGMT subnet
  • Apply: line vty 0 4 → access-class 10 in

NAT ACL: NAT-INTERNAL

  • Permit every internal /20 summary
  • Miss any one → partial internet outage
IPv6 Dual-Stack

Allocation

  • Aggregate: 2001:db8::/32
  • HQ: 2001:db8:1::/48
  • Branch: 2001:db8:2::/48
  • DC: 2001:db8:3::/48
  • WAN: 2001:db8:ffff::/48 → /64 per link

Config

  • ipv6 unicast-routing on every L3 device
  • Dual-stack on all 3 WAN /30s + HQ-USERS SVI
  • HQ-USERS: SLAAC (RAs enabled)
  • OSPFv3 process 1 area 0 on WAN + Loopback0

Phase 2

Chaos — 10 Seeded Misconfigs

60 min target

Save meridian-phase1-golden.pkt → meridian-phase2-chaos.pkt. Have a partner (or an LLM) seed the 10 misconfigs below into your chaos copy. You get 60 minutes with show, ping, traceroute, and sparing debug. Keep a log: hypothesis → command → result.

Read this before starting: symptoms overlap intentionally. One misconfig can produce two complaints. One complaint below is not a misconfig at all — it's policy working as designed. Distinguishing "broken" from "enforced" is half the exercise.
Symptom Report — What Users Called In
  • 01"The Denver branch can't print to the HQ printer." BR-USERS PC can ping an HQ user PC, but not the HQ printer. Printer-specific failure?
  • 02"Wi-Fi freezes every 30 seconds." VLAN 10 PC reports intermittent Request timed out in 30-second waves.
  • 03"DC lost internet this morning." VLAN 60 server → 8.8.8.8 fails. VLAN 40 BR PC → 8.8.8.8 works fine.
  • 04"Branch office is down, nobody's getting addresses." New BR-USERS PC → no DHCP lease. Statically-addressed PC works. BR-GUEST gets DHCP fine.
  • 05"I can't SSH into CORE1 from my laptop." From VLAN 10 → can ping CORE1 but SSH refused. From HQ-MGMT → SSH works.
  • 06"BR is routing through DC to get to HQ — long way around." Traceroute BR→HQ shows BR-RTR → DC-RTR → HQ-RTR. HQ↔BR link shows up/up.
  • 07"Po3 bundle is half-bandwidth." One member shows D (down) in show etherchannel summary. Cable is plugged in, link LED green.
  • 08"Clocks on BR and DC look wrong." BR-RTR + DC-RTR stratum 16 (unsynchronized). NTP key matches HQ — you verified twice.
  • 09"Guest Wi-Fi can see a printer it shouldn't." BR-GUEST PC → HQ printer: succeeds. BR-GUEST PC → HQ user: fails. The ACL is… partially working?
  • 10"ACC1 Fa0/15 keeps dying." Port went err-disabled. New laptop — dies in seconds. The port was fine yesterday.
The 10 Seeded Misconfigs (Reveal After Attempting)
FAULT 1 Summary route too broad (causes Symptom #1 AND #3)
Seed
DIST1
DIST1(config)# ip prefix-list HQ-SUMMARY seq 10 permit 10.0.0.0/21
! Too broad — /21 covers 10.0.0.0–10.0.7.255 AND bleeds toward DC subnets
Why it fools you
Produces TWO complaints (#1 Denver can't print + #3 DC lost internet) that look like separate bugs. One root cause, two blast radii. Classic red herring.
Diagnose
traceroute from BR-USERS to HQ-PRINTERS → dies at DIST1

show ip route 10.0.8.0 on HQ-RTR → too-specific match via wrong interface
DIST1 — Fix
DIST1(config)# no ip prefix-list HQ-SUMMARY seq 10 permit 10.0.0.0/21
DIST1(config)# ip prefix-list HQ-SUMMARY seq 10 permit 10.0.0.0/20
Summaries must be exactly as wide as the site — not narrower, not wider. /20 matches the HQ allocation; /21 lops off half the site AND bleeds toward DC.
FAULT 2 Native VLAN mismatch — 30s wave is a coincidence
Seed
DIST2 — trunk to ACC1
DIST2(config-if)# switchport trunk native vlan 1
! ACC1 side is still native 999
Why it fools you
The "30-second freeze" feels like STP (which IS 30 seconds). You'll waste time on STP. Actually: CDP-native-mismatch log floods correlate with retry timers. Look for %CDP-4-NATIVE_VLAN_MISMATCH, not STP events.
Diagnose
show interfaces trunk — "Native vlan" column differs between sides

show logging | include NATIVE_VLAN
DIST2 — Fix
DIST2(config-if)# switchport trunk native vlan 999
Both sides of a trunk must agree on the native VLAN. Mismatch sends untagged frames to the wrong VLAN — security hole AND latency source.
FAULT 3 NAT ACL missing the DC summary
Seed
HQ-RTR
HQ-RTR(config)# ip access-list extended NAT-INTERNAL
HQ-RTR(config-ext-nacl)# no permit ip 10.0.32.0 0.0.15.255 any
! DC /20 removed — DC hosts hit NAT, get denied, drop
Why it fools you
Combines with Fault 1 to make "DC lost internet" look like routing. It's actually NAT refusing to translate DC source IPs.
Diagnose
show ip access-lists NAT-INTERNAL

debug ip nat while pinging from DC-SERVERS — no translation entry created
HQ-RTR — Fix
HQ-RTR(config)# ip access-list extended NAT-INTERNAL
HQ-RTR(config-ext-nacl)# permit ip 10.0.32.0 0.0.15.255 any
NAT translates what the ACL tells it to. Miss any subnet = partial outage. show ip nat translations stays empty for affected sources — NAT never happened.
FAULT 4 DHCP snooping trust missing on uplink
Seed
BR-SW1 — uplink to BR-RTR
BR-SW1(config-if)# no ip dhcp snooping trust
! Snooping enabled + uplink untrusted = DHCP offers dropped at ingress
Why it fools you
BR-GUEST gets DHCP fine (snooping isn't enabled on VLAN 50) — so the "DHCP is broken" theory feels wrong. Actually: snooping is only broken for the VLANs it's enabled on.
Diagnose
show ip dhcp snooping — lists trusted interfaces; uplink missing

show ip dhcp snooping statistics — dropped OFFER count climbing
BR-SW1 — Fix
BR-SW1(config)# interface gi0/1
BR-SW1(config-if)# ip dhcp snooping trust
DHCP snooping assumes access ports are hostile and uplinks are safe. Forget to trust the uplink = every reply from the server gets dropped before reaching clients.
FAULT 5 NO MISCONFIG — policy working as designed
What the user reported
"I can't SSH to CORE1 from my laptop." From VLAN 10 → SSH refused. From HQ-MGMT → SSH works.
Why it fools you
It looks broken, so you start hunting. The vty access-class 10 permits only MGMT subnets. VLAN 10 is supposed to be blocked. This trains the reflex this lab really cares about: distinguish broken from enforced. Most real incidents are user confusion, not bugs.
How to verify
show access-lists 10 — list the permits

show line vty 0 4 — confirm access class 10 in applied

If both match intended policy → document "working as designed" and move on. Don't "fix" it.
Fix: none. This is policy. If you spent time "fixing" it, you weakened your security. Worth it to cultivate the instinct "verify the policy before you hunt a bug."
FAULT 6 OSPF hello/dead timer mismatch (HQ↔BR)
Seed
HQ-RTR — WAN to BR
HQ-RTR(config-if)# ip ospf hello-interval 5
HQ-RTR(config-if)# ip ospf dead-interval 20
! BR-RTR side still at default 10/40
Why it fools you
Physical link is up/up. You'll blame the cable. Actually: adjacency never forms — OSPF requires matching timers. BR reroutes via DC, which looks like OSPF "working" — the triangle hides it.
Diagnose
show ip ospf neighbor on HQ-RTR — BR neighbor missing or INIT

show ip ospf interface gi0/0/0 both sides — compare timers
HQ-RTR — Fix
HQ-RTR(config-if)# no ip ospf hello-interval
HQ-RTR(config-if)# no ip ospf dead-interval
Remove the overrides to restore defaults (10/40), OR set matching values on both sides. CCNA's single most tested OSPF troubleshooting pattern.
FAULT 7 Po3 LACP mode mismatch
Seed
DIST1 vs ACC2
DIST1(config-if-range)# channel-group 3 mode active
! ACC2 side configured mode 'on' (static, not LACP)
Why it fools you
Link lights green both sides. Members up. Bundle doesn't form. show etherchannel summary shows members with D flag.
Diagnose
show etherchannel summary — Po3 column s (suspended) or members D (down)

show etherchannel port-channel — compare "Protocol" column (LACP vs — )
ACC2 — Fix
ACC2(config-if-range)# no channel-group 3 mode on
ACC2(config-if-range)# channel-group 3 mode active
Compatible modes: active/active, active/passive, on/on. Mixing active with on = no bundle.
FAULT 8 NTP key configured but NOT trusted
Seed
BR-RTR
BR-RTR(config)# ntp authentication-key 1 md5 meridian-ntp
BR-RTR(config)# ntp server 10.0.49.1 key 1
! ntp trusted-key 1 is MISSING
Why it fools you
You re-type the key. You verify it matches. You check the path. You suspect an ACL on UDP 123. None of that matters — without ntp trusted-key, the key is ignored and auth silently fails. Stratum stays at 16.
Diagnose
show ntp associations detail — "not authenticated"

show running | include ntp — compare against HQ; trusted-key line missing
BR-RTR + DC-RTR — Fix
BR-RTR(config)# ntp trusted-key 1
NTP auth is 3-part: define key, bind to server, trust the key. Miss step 3 and Cisco silently rejects auth. Classic trap.
FAULT 9 ACL line order inverted (BR-GUEST → printers passes)
Seed
HQ-RTR
HQ-RTR(config)# ip access-list extended BLOCK-GUEST-INTERNAL
 10 permit ip any any
 20 deny ip 10.0.17.0 0.0.0.127 10.0.0.0 0.0.15.255
! Permit line 10 hits first — deny never evaluated
Why it fools you
Guest → HQ-USERS gets blocked by a different ACL upstream, so it looks like half the rule works. Actually, this ACL is bypassed entirely — the "working" case is accidental.
Diagnose
show ip access-lists BLOCK-GUEST-INTERNAL — read line by line

ACL hit counters: permit line has high count, deny line has zero
HQ-RTR — Fix (reorder)
HQ-RTR(config)# ip access-list extended BLOCK-GUEST-INTERNAL
HQ-RTR(config-ext-nacl)# no 10
HQ-RTR(config-ext-nacl)# no 20
HQ-RTR(config-ext-nacl)# 10 deny ip 10.0.17.0 0.0.0.127 10.0.0.0 0.0.15.255
HQ-RTR(config-ext-nacl)# 20 deny ip 10.0.17.0 0.0.0.127 10.0.32.0 0.0.15.255
HQ-RTR(config-ext-nacl)# 30 permit ip any any
ACLs are first-match. Always deny-specific before permit-general. The moment you have permit ip any any above a deny, the deny is dead code.
FAULT 10 Cable loop + BPDU Guard = err-disable
Seed
ACC1 — physical cable
! Someone plugged a cable from Fa0/15 to Fa0/16 (both portfast+BPDUguard)
! Fa0/15 receives a BPDU from Fa0/16 → BPDU Guard trips → err-disable
Why it fools you
Three laptops on the port all "break" → you blame the laptops, the NIC, the driver. None of them are the problem. The protection feature fires the instant any device plugged in triggers a loop on the adjacent port.
Diagnose
show interface status err-disabled — reason: bpduguard

show logging | include BPDU — %SPANTREE-2-BLOCK_BPDUGUARD

Walk the cabling: Fa0/15 ↔ Fa0/16 is a loop
ACC1 — Fix
ACC1# ! Physically remove the rogue cable, then:
ACC1(config)# interface fa0/15
ACC1(config-if)# shutdown
ACC1(config-if)# no shutdown
ACC1(config)# errdisable recovery cause bpduguard
ACC1(config)# errdisable recovery interval 300
BPDU Guard is doing its job — killed the port to prevent a loop from taking down the L2 domain. Fix is physical (pull the cable) + procedural (auto-recovery so transients don't require manual intervention).
Post-Mortem (Required Deliverable)
The real takeaway isn't the 10 fixes. It's the reflex table you build afterward. For each fault, write down: symptom → first thing you checked → what the root cause actually was → what you'd check first next time. The last column is what you're training.
Phase 3

Resilience — 5 Failure Drills

~90 min · stopwatch

Induce failures. Measure recovery. The network isn't "good" because it converges — it's good because you know how long convergence takes. Work on a fresh chaos copy. Continuous-ping during each drill; record seconds to recover and packets lost.

Drill 1 — WAN Link Failure (HQ ↔ BR)

Target: ≤25s Measures: OSPF reconvergence
Setup ping: BR-USERS PC → HQ-USERS PC continuous.
Induce: HQ-RTR(config-if)# shutdown on Gi0/0/0.
Expected: dead timer fires (10s default or 20s if you set 5/20), route rebuilds via BR→DC→HQ triangle, ping resumes ~15–25s later.
One-way positive + negative: ping resumes (✓), AND BR-GUEST still cannot reach HQ-USERS via the new path (ACL still enforced).

Drill 2 — Core Switch Failure (HSRP + STP)

Target: ≤6s Measures: HSRP failover + STP re-election
Setup ping: VLAN 10 PC → HSRP VIP continuous.
Induce: CORE1# reload.
Expected: HSRP (hello 1s / hold 3s) transitions CORE2 to Active for VLAN 10/30/99. STP re-roots. Ping resumes ~4–6s.
Failback watch: when CORE1 returns, does preempt delay 30 prevent double-blackhole during OSPF re-convergence?

Drill 3 — DHCP Server Pool Failure

No target — graceful degradation Measures: existing leases vs new
Setup: One existing BR-USERS PC with a lease; one fresh PC about to plug in.
Induce: HQ-RTR(config)# no ip dhcp pool BR-USERS.
Expected: existing PC keeps working until T1 renewal. New PC sends DHCPDISCOVER, relay forwards, server silently drops (no pool), client times out to APIPA 169.254.x.x.
Lesson: "DHCP is broken" is ambiguous — could be snooping, relay, or server pool. Know how to distinguish.

Drill 4 — LACP Member Pull

Target: 0 packets lost Measures: graceful bundle degradation
Setup ping: VLAN 10 PC → HSRP VIP continuous.
Induce: DIST1(config)# interface fa0/1; shutdown (one Po1 member).
Expected: LACP removes the member, Po1 stays SU, member count 2→1, bandwidth halves, zero packet loss. LACP is designed for this.

Drill 5 — ISP Link Down (Default Route)

Target: internal 0 loss Measures: default-route behavior
Dual ping streams: BR→HQ (internal, should stay up) + BR→8.8.8.8 (internet, should fail cleanly).
Induce: HQ-RTR(config-if)# shutdown on Gi0/2 (ISP).
With default-information originate always: black-hole at HQ-RTR.
Without always: OSPF withdraws the default, remote sites get ICMP unreachables, clean failure.
Document which you chose and why.
Phase 4 (stretch)

Automation — NetDevOps bridge

1–2 days · post-CCNA

CCNA blueprint touches REST/JSON/Ansible conceptually. This phase makes it concrete: audit.py pulls all 12 configs in parallel, healthcheck.py parses show outputs and finds every Phase 2 misconfig in ~15 seconds. Portfolio-worthy.

Repo layout

  • inventory.yaml — devices, IPs, roles, creds from .env
  • audit.py — netmiko + concurrent.futures, parallel SSH, snapshots
  • healthcheck.py — parses show output, JSON report
  • diff.py — structured diff vs golden snapshot
  • drill.py — scripted chaos + recovery timing
  • Makefile — make audit, make healthcheck, make diff

Environment

  • Mock mode: library of canned show outputs — prove parsing logic
  • Cisco CML: free personal tier, real IOS, real SSH — recommended
  • GNS3: alternative if you have IOS images
  • Packet Tracer's SSH is fake — can't script it directly

Healthcheck checks

  • show ip ospf neighbor — all FULL
  • show standby brief — Active/Standby per VLAN
  • show etherchannel summary — all SU, no D
  • show ntp status — Clock synchronized
  • Should light up on every Phase 2 misconfig

Definition of done

  • make audit pulls 12 devices in parallel, zero errors
  • make healthcheck correctly IDs every Phase 2 misconfig
  • make diff = empty against golden
  • 5+ drill runs recorded in drill-history.csv
