Step-by-Step Build
01 Build the topology in Packet Tracer

Place an ISR4321 router and a Server. Connect Server1 Fa0 to R1 GigabitEthernet0/0/0 with a Copper Cross-Over cable. The server represents a REST API client (like curl or Postman).

02 Configure the router

Configure R1 with a hostname, interface IP, description, and enable the interface. Pay close attention to every command — in the next section you'll see this exact config as a JSON payload.

R1
Router> enable
Router# configure terminal
Router(config)# hostname R1
R1(config)# interface GigabitEthernet0/0/0
R1(config-if)# description WAN Link
R1(config-if)# ip address 10.0.0.1 255.255.255.0
R1(config-if)# no shutdown
R1(config-if)# end
03 Configure the server

Click Server1 → Desktop → IP Configuration. Set IP 10.0.0.10, subnet 255.255.255.0, gateway 10.0.0.1.

04 Verify connectivity

Ping from Server1 to R1 to confirm the link is up.

R1
R1# show ip interface brief
! GigabitEthernet0/0/0 should show up/up with 10.0.0.1
R1# show running-config interface GigabitEthernet0/0/0
! Review every line — you'll see these as JSON fields next
Bridge to APIs

The show running-config output you just saw is what a RESTCONF GET request returns — but as structured JSON instead of CLI text. Every line maps to a JSON field. The rest of this lab teaches you that mapping.

05 Save your configuration
R1
R1# copy running-config startup-config

Analysis Exercises
01 HTTP Verb Mapping to IOS Equivalents

RESTCONF maps standard HTTP methods to network configuration operations. Study this table — the CCNA exam frequently asks which HTTP method corresponds to a given action.

HTTP Verb IOS Equivalent Behavior
GET show commands Read-only retrieval. Does not modify any configuration.
PUT Replace a config section Replaces the entire resource. Like overwriting a config block — existing values not in the payload are removed.
POST Create new resource Creates a new entry (e.g., add a new interface or route). Fails if the resource already exists.
PATCH Modify existing resource Merges changes into the existing config. Like adding lines to an existing config section without removing others.
DELETE Remove config Removes the specified resource. Equivalent to using no on a config command.

Key insight: GET is the safe, read-only verb. PUT is destructive — it replaces, not merges. Use PATCH when you want to add to existing configuration without erasing it.

02 JSON Payload Analysis — Interface Configuration

Below is a realistic RESTCONF JSON response for a router interface. Study it and answer the mapping questions that follow.

RESTCONF GET response — ietf-interfaces
{
  "ietf-interfaces:interface": {
    "name": "GigabitEthernet1",
    "description": "WAN Link",
    "type": "iana-if-type:ethernetCsmacd",
    "enabled": true,
    "ietf-ip:ipv4": {
      "address": [
        {
          "ip": "10.0.0.1",
          "prefix-length": 24
        }
      ]
    }
  }
}

Mapping questions — think through each before reading the answer:

What IOS command does "enabled": true correspond to? Answer: no shutdown — the interface is administratively up. If enabled were false, it maps to shutdown.

What does "prefix-length": 24 map to? Answer: subnet mask 255.255.255.0 — RESTCONF uses CIDR prefix length instead of dotted-decimal subnet masks.

What IOS command does the entire payload represent? Answer: interface GigabitEthernet1 / description WAN Link / ip address 10.0.0.1 255.255.255.0 / no shutdown

03 RESTCONF URL Structure

RESTCONF URLs follow a predictable pattern. Break down this example URL and understand each segment — exam questions often ask what a specific part of the URL represents.

RESTCONF URL — retrieve GigabitEthernet1 config
https://192.168.1.1/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1
URL Segment Meaning
/restconf/data/ Datastore path — this is where running/candidate configuration lives. All config reads/writes go through /data/.
ietf-interfaces YANG module name (RFC 7223). The part before the colon is always the module that defines this data model.
:interfaces Top-level container within the YANG module. The colon separates module from data node.
/interface=GigabitEthernet1 List key — identifies a specific instance within the interface list. The value after = is the key value (the interface name).

Pattern to memorize: /restconf/data/[module]:[container]/[list]=[key]

04 YANG Model Navigation — Paths to IOS Equivalents

YANG (Yet Another Next Generation) is a data modeling language (RFC 6020/7950) that defines the structure, types, and relationships of network configuration data. It is not a protocol — NETCONF and RESTCONF are the protocols that use YANG models.

YANG Path IOS Equivalent
ietf-interfaces:interfaces/interface/name Interface name (e.g., GigabitEthernet1)
ietf-interfaces:interfaces/interface/enabled Admin state — no shutdown (true) / shutdown (false)
ietf-ip:ipv4/address/ip IPv4 address in the ip address command
ietf-ip:ipv4/address/prefix-length Subnet mask length (CIDR notation)
ietf-routing:routing/control-plane-protocols Routing protocol configuration block

NETCONF vs. RESTCONF comparison:

Feature NETCONF RESTCONF
Transport SSH (port 830) HTTPS (port 443)
Data format XML only JSON or XML
Operations get, get-config, edit-config, lock, unlock, commit HTTP verbs (GET, PUT, POST, PATCH, DELETE)
Transactions Full transactional support (lock/commit) No native transaction support
Best for Complex, transactional bulk changes Simple reads/writes, REST tooling (curl, Postman)

Both use the same YANG data models — the model describes what can be configured, not how the protocol transports it.

05 REST API Authentication Types

REST APIs require authentication to control access. The CCNA expects you to recognize the three common methods and understand when each is used.

Auth Type How It Works Security Notes
Basic Auth Base64-encoded username:password sent in the Authorization: Basic header Simple but credentials are sent with every request. Only safe over HTTPS.
Token / Bearer API key or token sent in the Authorization: Bearer header More secure — tokens can expire and be revoked without changing the password.
OAuth 2.0 Delegated authorization. Client obtains a token from an auth server, then uses it for API calls. Most complex but most secure. Supports scoped permissions and token refresh.
curl — Basic Auth to RESTCONF endpoint
$ curl -u admin:cisco https://192.168.1.1/restconf/data/ietf-interfaces:interfaces
# -u flag encodes admin:cisco as base64 and sends it in the Authorization header
# Equivalent header: Authorization: Basic YWRtaW46Y2lzY28=

CCNA takeaway: Cisco RESTCONF typically uses Basic Auth over HTTPS. Know that -u user:pass in curl maps to the Authorization: Basic header, and that tokens provide better security because they expire.

06 JSON vs XML — Same Data, Different Format

The same YANG model data can be encoded as JSON or XML. Compare the two representations of the interface configuration from Exercise 2 — both describe the exact same config.

JSON (RESTCONF)
{
  "ietf-interfaces:interface": {
    "name": "GigabitEthernet1",
    "description": "WAN Link",
    "enabled": true,
    "ietf-ip:ipv4": {
      "address": [{
        "ip": "10.0.0.1",
        "prefix-length": 24
      }]
    }
  }
}
XML (NETCONF)
<interface xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
  <name>GigabitEthernet1</name>
  <description>WAN Link</description>
  <enabled>true</enabled>
  <ipv4 xmlns="urn:ietf:params:xml:ns:yang:ietf-ip">
    <address>
      <ip>10.0.0.1</ip>
      <prefix-length>24</prefix-length>
    </address>
  </ipv4>
</interface>
Feature JSON XML
Structure key: value pairs, curly braces {} <tag>value</tag> elements
Used by RESTCONF (default) NETCONF (exclusively)
Human readability More compact, easier to scan More verbose, self-describing

Key fact: NETCONF uses XML exclusively. RESTCONF supports both JSON and XML but defaults to JSON. Both encode the same underlying YANG data model — the format is just the serialization, not the data itself.

Configuration Management — Ansible & Terraform (Objective 6.6)
A Ansible

Agentless — uses SSH to connect to targets. No software needs to be installed on managed devices. Push model: the control node pushes configuration to devices. Configurations are written in YAML files called playbooks. Ansible is idempotent — running the same playbook twice produces the same result (it won't re-apply changes that are already present).

Cisco modules: ios_config (push config lines), ios_command (run show commands).

Ansible Playbook — configure hostname
---
- name: Configure hostname
  hosts: routers
  tasks:
    - name: Set hostname
      cisco.ios.ios_config:
        lines:
          - hostname R1
B Terraform

Declarative infrastructure-as-code. Uses HCL (HashiCorp Configuration Language). Manages state via a state file that tracks what infrastructure currently exists. Workflow: terraform plan (preview changes) → terraform apply (execute changes). Terraform provisions infrastructure (Day 0/1 — creating VMs, networks, cloud resources), not device configuration (Day 2 — that's Ansible's job).

C Comparison — Ansible vs Terraform vs Python
Tool Language Model Agentless? Primary Use
Ansible YAML Push Yes Configuration management
Terraform HCL Declarative Yes Infrastructure provisioning
Python scripts Python Custom Varies Automation scripts
CCNA exam note: The CCNA does NOT require you to write Ansible playbooks or Terraform configs. You must recognize what each tool does, how it connects (SSH vs API), and when to use one vs the other. Ansible = config management via SSH + YAML. Terraform = infrastructure provisioning via HCL + state files.
