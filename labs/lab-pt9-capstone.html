The Challenge

This is the capstone. Three sites, fifteen devices, twelve objectives — every major CCNA domain in a single build. You know the commands. Now prove it. Each step tells you what to configure, not how. A hint with the CLI commands is hidden behind a toggle — use it only if you get stuck. If you can complete this lab without opening a single hint, you are exam-ready.

Objectives
  • 01Build the full topology and cable all devices (3 routers, 3 switches, 4 PCs, 1 server)
  • 02Configure VLANs 10, 20, 30, 99 on all switches with 802.1Q trunking
  • 03Configure Router-on-a-Stick (subinterfaces) on R1 and R2 for inter-VLAN routing
  • 04Build a 2-port LACP EtherChannel (Po1) between SW1 and SW2
  • 05Set SW1 as STP root bridge for all VLANs; enable PortFast + BPDU Guard on access ports
  • 06Configure OSPF Area 0 on the R1-R2 WAN link and advertise all LAN subnets
  • 07Configure DHCP pools on R1 for HQ VLANs 10 and 20 (with relay if needed)
  • 08Configure PAT (NAT overload) on R1 for internet access through R3
  • 09Create a named extended ACL blocking Engineering (VLAN 20) from reaching Servers (VLAN 30)
  • 10Enable port security (sticky MAC, max 1, violation shutdown) on all access ports
  • 11Enable DHCP snooping on HQ switches
  • 12Configure SSH-only management access on all routers (disable Telnet)

Step-by-Step Build
01 Build the topology and cable all devices

Place 3 routers (ISR4321), 3 switches (2960), 4 PCs, and 1 server in Packet Tracer. Cable them according to the topology diagram. Use Copper Straight-Through for PC/server-to-switch connections, Copper Cross-Over for switch-to-switch (the EtherChannel pair), and Copper Cross-Over for router-to-router WAN links. The trunk from each router to its local switch uses a straight-through cable.

Checkpoint: All links should show orange dots (not yet configured) or green dots. No red X marks — those indicate a wrong cable type or wrong port.

02 Configure VLANs and trunking on all switches

On every switch (SW1, SW2, SW3), create VLANs 10 (Sales), 20 (Engineering), 30 (Servers), and 99 (Native). Assign the access ports to their correct VLANs based on the addressing table. Configure the uplink trunk ports: SW1 Gi0/1 to R1, and SW3 Fa0/24 to R2. Set native VLAN 99 on all trunks and limit allowed VLANs to 10, 20, 30, 99. Disable DTP on every trunk with nonegotiate. Do not configure the EtherChannel ports as trunks yet — that happens in Step 04.

03 Configure Router-on-a-Stick on R1 and R2

On R1, create subinterfaces on Gi0/0/0 for VLANs 10, 20, and 30. Each subinterface must specify 802.1Q encapsulation with the correct VLAN ID and be assigned the gateway IP from the addressing table. Do not forget to no shutdown the physical interface. On R2, do the same on Gi0/0/0 for VLANs 10 and 20. Also configure the WAN interfaces (R1 Gi0/0/1, R2 Gi0/0/1) and the ISP link (R1 Gi0/0/2) with their IP addresses.

04 Build a 2-port LACP EtherChannel between SW1 and SW2

Bundle Fa0/23 and Fa0/24 on both SW1 and SW2 into Port-channel 1 using LACP. One side should be active, the other active or passive (active-active is the most common production config). After the channel is up, configure Po1 as a trunk with native VLAN 99 and allowed VLANs 10, 20, 30, 99. Remember: configure the port-channel interface for trunk settings, not the individual member ports.

Show hint — EtherChannel config
SW1
SW1(config)# interface range fastethernet 0/23 - 24
SW1(config-if-range)# channel-group 1 mode active
SW1(config-if-range)# exit
SW1(config)# interface port-channel 1
SW1(config-if)# switchport mode trunk
SW1(config-if)# switchport trunk native vlan 99
SW1(config-if)# switchport trunk allowed vlan 10,20,30,99
SW1(config-if)# switchport nonegotiate
SW1(config-if)# exit
SW2
SW2(config)# interface range fastethernet 0/23 - 24
SW2(config-if-range)# channel-group 1 mode active
SW2(config-if-range)# exit
SW2(config)# interface port-channel 1
SW2(config-if)# switchport mode trunk
SW2(config-if)# switchport trunk native vlan 99
SW2(config-if)# switchport trunk allowed vlan 10,20,30,99

05 STP root bridge, PortFast, and BPDU Guard

Make SW1 the STP root bridge for all VLANs by lowering its bridge priority. On all three switches, enable PortFast and BPDU Guard on every access port (ports connecting to PCs and Server1). PortFast skips the STP listening/learning states on edge ports. BPDU Guard shuts down a port if it receives a BPDU — preventing rogue switches from being plugged in.

Show hint — STP + PortFast config
SW1 — Root Bridge
SW1(config)# spanning-tree vlan 10,20,30,99 root primary
! --- PortFast + BPDU Guard on access port ---
SW1(config)# interface fastethernet 0/1
SW1(config-if)# spanning-tree portfast
SW1(config-if)# spanning-tree bpduguard enable
SW1(config-if)# exit
SW2 — Access Ports
SW2(config)# interface range fastethernet 0/1 - 2
SW2(config-if-range)# spanning-tree portfast
SW2(config-if-range)# spanning-tree bpduguard enable
SW2(config-if-range)# exit
SW3 — Access Ports
SW3(config)# interface range fastethernet 0/1 - 2
SW3(config-if-range)# spanning-tree portfast
SW3(config-if-range)# spanning-tree bpduguard enable
SW3(config-if-range)# exit
06 Configure OSPF Area 0 on R1 and R2

Enable OSPF process 1 on both R1 and R2. Advertise the WAN link (10.0.12.0/30) and all LAN subnets into Area 0 using network statements. Set a router-id on each router. Make all LAN-facing interfaces (subinterfaces) passive — only the WAN interface should form adjacencies. On R1, also configure a default route to R3 (203.0.113.2) and redistribute it into OSPF so R2 learns the internet path.

Show hint — R1 OSPF config
R1
R1(config)# router ospf 1
R1(config-router)# router-id 1.1.1.1
R1(config-router)# network 10.0.12.0 0.0.0.3 area 0
R1(config-router)# network 10.10.10.0 0.0.0.255 area 0
R1(config-router)# network 10.10.20.0 0.0.0.255 area 0
R1(config-router)# network 10.10.30.0 0.0.0.255 area 0
R1(config-router)# passive-interface gigabitethernet 0/0/0.10
R1(config-router)# passive-interface gigabitethernet 0/0/0.20
R1(config-router)# passive-interface gigabitethernet 0/0/0.30
R1(config-router)# default-information originate
R1(config-router)# exit
! --- Default route to ISP ---
R1(config)# ip route 0.0.0.0 0.0.0.0 203.0.113.2

07 Configure DHCP on R1 for HQ VLANs

Create two DHCP pools on R1 — one for VLAN 10 (10.10.10.0/24) and one for VLAN 20 (10.10.20.0/24). Exclude the first 10 addresses in each range for static assignments and the gateway. Set the default gateway and DNS server (use Server1 at 10.10.30.10 as DNS). After configuring, go to PC1 and PC2 in Packet Tracer, switch them to DHCP mode, and verify they receive addresses.

R1(config)# ip dhcp pool VLAN10-SALES
R1(dhcp-config)# network 10.10.10.0 255.255.255.0

08 Configure PAT (NAT overload) on R1

Configure NAT overload (PAT) on R1 so all internal subnets can reach the internet through the single public IP on Gi0/0/2 (203.0.113.1). You need three things: (1) an ACL that matches all internal traffic (10.0.0.0/8 covers everything), (2) mark the inside and outside interfaces, and (3) tie the ACL to NAT overload on the outside interface. Both HQ and Branch traffic should be NATted — Branch traffic arrives via OSPF and already passes through R1 on the way to R3.

Show hint — NAT/PAT config
R1
R1(config)# access-list 1 permit 10.0.0.0 0.255.255.255
! --- Mark interfaces ---
R1(config)# interface gigabitethernet 0/0/0.10
R1(config-subif)# ip nat inside
R1(config-subif)# exit
R1(config)# interface gigabitethernet 0/0/0.20
R1(config-subif)# ip nat inside
R1(config-subif)# exit
R1(config)# interface gigabitethernet 0/0/0.30
R1(config-subif)# ip nat inside
R1(config-subif)# exit
R1(config)# interface gigabitethernet 0/0/1
R1(config-if)# ip nat inside
R1(config-if)# exit
R1(config)# interface gigabitethernet 0/0/2
R1(config-if)# ip nat outside
R1(config-if)# exit
! --- PAT overload ---
R1(config)# ip nat inside source list 1 interface gigabitethernet 0/0/2 overload
09 Create a named extended ACL blocking Engineering from Servers

Create a named extended ACL called BLOCK-ENG-TO-SERVERS that denies all IP traffic from the HQ Engineering subnet (10.10.20.0/24) to the Servers subnet (10.10.30.0/24), then permits everything else. Apply it inbound on R1's Gi0/0/0.20 subinterface — this is closest to the source, which is the correct placement for an extended ACL. Think about why you would NOT apply it outbound on the Gi0/0/0.30 subinterface instead.

Show hint — ACL config
R1
R1(config)# ip access-list extended BLOCK-ENG-TO-SERVERS
R1(config-ext-nacl)# deny ip 10.10.20.0 0.0.0.255 10.10.30.0 0.0.0.255
R1(config-ext-nacl)# permit ip any any
R1(config-ext-nacl)# exit
R1(config)# interface gigabitethernet 0/0/0.20
R1(config-subif)# ip access-group BLOCK-ENG-TO-SERVERS in
R1(config-subif)# exit
ACL Placement Rule

Extended ACLs go close to the source. Standard ACLs go close to the destination. Since this ACL specifies both source and destination, it is extended and belongs on the interface closest to the Engineering subnet — R1's Gi0/0/0.20 inbound.

10 Enable port security on all access ports

On every switch access port (SW1 Fa0/1, SW2 Fa0/1-2, SW3 Fa0/1-2), configure port security with: maximum 1 MAC address, sticky learning, and violation mode shutdown. This locks each port to the first device that connects. If someone unplugs a PC and plugs in a rogue device, the port shuts down.

Show hint — port security config
SW1
SW1(config)# interface fastethernet 0/1
SW1(config-if)# switchport port-security
SW1(config-if)# switchport port-security maximum 1
SW1(config-if)# switchport port-security mac-address sticky
SW1(config-if)# switchport port-security violation shutdown
SW1(config-if)# exit
SW2
SW2(config)# interface range fastethernet 0/1 - 2
SW2(config-if-range)# switchport port-security
SW2(config-if-range)# switchport port-security maximum 1
SW2(config-if-range)# switchport port-security mac-address sticky
SW2(config-if-range)# switchport port-security violation shutdown
SW2(config-if-range)# exit
SW3
SW3(config)# interface range fastethernet 0/1 - 2
SW3(config-if-range)# switchport port-security
SW3(config-if-range)# switchport port-security maximum 1
SW3(config-if-range)# switchport port-security mac-address sticky
SW3(config-if-range)# switchport port-security violation shutdown
SW3(config-if-range)# exit
11 Enable DHCP snooping on HQ switches

Enable DHCP snooping globally on SW1 and SW2 for VLANs 10 and 20. Mark the uplink/trunk ports as trusted (the port facing R1, and the EtherChannel Po1). All access ports remain untrusted by default — they will only allow DHCP requests, not rogue DHCP offers. This prevents a rogue DHCP server from handing out fake gateway addresses.

Show hint — DHCP snooping config
SW1
SW1(config)# ip dhcp snooping
SW1(config)# ip dhcp snooping vlan 10,20
! --- Trust the trunk to R1 ---
SW1(config)# interface gigabitethernet 0/1
SW1(config-if)# ip dhcp snooping trust
SW1(config-if)# exit
! --- Trust the EtherChannel ---
SW1(config)# interface port-channel 1
SW1(config-if)# ip dhcp snooping trust
SW1(config-if)# exit
SW2
SW2(config)# ip dhcp snooping
SW2(config)# ip dhcp snooping vlan 10,20
! --- Trust the EtherChannel ---
SW2(config)# interface port-channel 1
SW2(config-if)# ip dhcp snooping trust
SW2(config-if)# exit
12 Configure SSH-only management on all routers

On R1, R2, and R3: set a domain name, generate RSA keys (2048-bit), create a local user account, and configure the VTY lines to accept only SSH (no Telnet). Set SSH version 2. Use transport input ssh on VTY 0-4 and authenticate with the local database. Also set an enable secret and a console password.

Show hint — SSH config (apply to R1, R2, R3)
R1 (repeat for R2 and R3)
R1(config)# ip domain-name lab.local
R1(config)# crypto key generate rsa general-keys modulus 2048
R1(config)# ip ssh version 2
R1(config)# username admin privilege 15 secret Cisco123!
R1(config)# enable secret Class123!
R1(config)# line vty 0 4
R1(config-line)# transport input ssh
R1(config-line)# login local
R1(config-line)# exit
R1(config)# line console 0
R1(config-line)# password Cisco123!
R1(config-line)# login
R1(config-line)# exit
R1(config)# service password-encryption
13 Save all configurations

Save the running config to startup on every device (R1, R2, R3, SW1, SW2, SW3). Configure PC3 and PC4 with static IPs, and Server1 with its static IP and gateway. Set PC1 and PC2 to DHCP. Then save the Packet Tracer file (Ctrl+S). You are ready to verify.

Break/Fix Challenge

Your network is working. Now we break three things. Your job: find the fault and fix it using only show and debug commands. Do not look at the answers until you have identified the root cause yourself.

F1 OSPF adjacency lost

Symptom: R2 can no longer reach any HQ subnets. Cross-site pings fail. R1 shows no OSPF neighbors.

Diagnostic hint: Check show ip ospf interface on R2. Look at the WAN interface status.

Show cause
Root Cause

Someone added passive-interface gigabitethernet 0/0/1 to R2's OSPF process. A passive interface does not send or receive OSPF Hello packets — the adjacency drops immediately.

Show fix
R2
R2(config)# router ospf 1
R2(config-router)# no passive-interface gigabitethernet 0/0/1
R2(config-router)# exit
F2 Inter-VLAN routing broken at HQ

Symptom: PC1 (VLAN 10) cannot ping PC2 (VLAN 20) or Server1 (VLAN 30) at HQ. PC1 can still ping its own gateway (10.10.10.1). The router subinterfaces show up/up.

Diagnostic hint: Check show interfaces trunk on SW1. Compare the native VLAN on the switch trunk to the encapsulation on R1's subinterfaces.

Show cause
Root Cause

The native VLAN on SW1's Gi0/1 trunk was changed to VLAN 10 instead of VLAN 99. Now VLAN 10 frames cross untagged, but R1's Gi0/0/0.10 subinterface expects tagged frames with VLAN 10. The mismatch causes VLAN 10 traffic to be misclassified and dropped by the router.

Show fix
SW1
SW1(config)# interface gigabitethernet 0/1
SW1(config-if)# switchport trunk native vlan 99
SW1(config-if)# exit
F3 Branch cannot reach the internet

Symptom: PC3 at the Branch can ping HQ subnets (10.10.10.x, 10.10.20.x) but cannot ping 8.8.8.8. Meanwhile, HQ PCs can ping 8.8.8.8 just fine.

Diagnostic hint: Run show ip nat translations on R1 after PC3 tries to ping 8.8.8.8. Check if Branch source addresses appear. Then check the NAT ACL with show access-lists 1.

Show cause
Root Cause

The NAT ACL (access-list 1) was changed to permit 10.10.0.0 0.0.255.255 — this only matches HQ subnets (10.10.x.x). Branch subnets (10.20.x.x) are not matched by the ACL, so their traffic is not translated by NAT and R3 drops the packets (R3 has no route back to 10.20.x.x).

Show fix
R1
R1(config)# no access-list 1
R1(config)# access-list 1 permit 10.0.0.0 0.255.255.255
