Why does the internet use packet switching?

Because reserving a dedicated circuit per conversation (like PSTN did) wastes capacity whenever traffic is bursty — which is almost always. Packet switching lets routers statistically multiplex many flows onto shared links, absorbing spikes in queues instead of blocking calls. The tradeoff: variable latency and occasional loss, which TCP compensates for at the endpoints.

Math

Core formulas

Total delay per hop

d = dtrans + dprop + dproc + dqueue

Four additive components. Transmission = packet size ÷ bandwidth. Propagation = distance ÷ speed of light (in medium).

Little's Law

L = λ × W

Average queue length (L) = arrival rate (λ) × average wait (W). Predicts when routers start dropping.

Transmission delay

dtrans = L ÷ R

Packet length (L bits) ÷ link rate (R bps). A 1500-byte packet on 100 Mbps = 120 µs.

  1. Which router turns red first? Why that one?
  2. What happens to average latency as queues fill?
  3. If you were the netops engineer, what would you change?
Activity 3 · Routing races

Swap algorithms

Keep all params constant. Run Shortest Path, then Flooding, then Link State.

  1. How many hops does each algorithm use to deliver?
  2. Which one would converge fastest if a link failed mid-run?
  3. Why don't we use flooding in production?

  1. Which transport "feels" more successful in metrics?
  2. What is TCP doing behind the scenes to hide loss?
  3. When would UDP be the correct choice anyway?

Watch out

Common mistakes

✗"Packet switching has no latency cost."

Reality: Queueing delay often dominates transmission delay. Under load, packets can wait longer in buffers than they spend on the wire.

✗"Circuit switching is obsolete."

Reality: SONET/SDH rings, dedicated lambda services, and some private WAN circuits still use circuit semantics — especially where guaranteed latency matters (broadcast TV, research nets).

✗"More bandwidth = faster network."

Reality: Bandwidth is throughput ceiling. Bufferbloat, loss, and bad queue management can make a 10 Gbps link feel worse than a well-tuned 100 Mbps one.

✗"Routers compute the path for every packet."

Reality: Routers do a fast-path lookup in their forwarding table (FIB). The heavy lifting — running Dijkstra or BGP best-path — happens in the control plane and only updates the FIB when topology changes.

✗"Flooding is dumb and never used."

Reality: Flooding is how OSPF propagates LSAs, how ARP works, how STP discovers the topology. It's bounded (by TTL or split-horizon rules), not dumb.

In practice

Real-world applications

☎️

Phone networks

PSTN was pure circuit. Modern cellular voice (VoLTE) rides IP packets but reserves QoS slots to preserve call quality.

📹

Video conferencing

Zoom, Meet, Teams use UDP-based RTP with selective retransmit. Pure TCP would create head-of-line blocking.

🏢

SD-WAN

Overlays multiple cheap links and picks the best per-packet based on loss/latency/jitter. Packet switching with policy.

📧

Email relay

Classic message switching — SMTP store-and-forward. Servers queue the full message until they can hand it off.

🎥

IPTV & broadcast

Multicast packet delivery with reserved VLANs or MPLS LSPs so critical streams never compete with internet traffic.

🛰️

Satellite relay

High-latency links force TCP tuning — window scaling, selective ACKs, TCP-proxies at the gateway.

🏭

Industrial control

PROFINET, EtherCAT, and deterministic Ethernet reserve time-slots — circuit-switching thinking applied to Ethernet.
