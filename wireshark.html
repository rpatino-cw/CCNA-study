Installation

Free and open source. Download from wireshark.org.

Windows
  1. Download .exe installer from wireshark.org
  2. Run installer — install Npcap when prompted (required for live capture)
  3. Launch Wireshark — select your active NIC to start capturing
macOS
  1. brew install --cask wireshark
  2. Or download .dmg from wireshark.org
  3. Grant network access when prompted
Linux
  1. sudo apt install wireshark
  2. Add your user to the wireshark group: sudo usermod -aG wireshark $USER
  3. Log out and back in
CCNA note: You won't install Wireshark on the exam. But you will be shown packet captures and asked to interpret them. Focus on reading captures, not operating the tool.

TCP Flags

Six flags you must know. They appear in every TCP segment header.

SYN
Synchronize. Starts a connection. Sets initial sequence number.
ACK
Acknowledge. Confirms receipt. Present in almost every segment after handshake.
FIN
Finish. Gracefully closes one direction of the connection.
RST
Reset. Abruptly terminates. "Connection refused" or "go away."
PSH
Push. Deliver data to app immediately, don't buffer.
URG
Urgent. Urgent pointer field is valid. Rarely used.

—— 3-Way Handshake (open) ————————————

Client → Server  [ACK]            Ack=301

—— Data Transfer ———————————————
Client → Server  [PSH, ACK]      Seq=101, Len=500
Server → Client  [ACK]            Ack=601

—— 4-Way Teardown (close) ——————————
Client → Server  [FIN, ACK]      Seq=601
Server → Client  [ACK]            Ack=602
Server → Client  [FIN, ACK]      Seq=800
Client → Server  [ACK]            Ack=801

How to Read a Capture

When the exam shows you a capture, follow this order every time.

Step 1: Check Source and Destination IPs

Who is talking to whom? Is the source internal (192.168.x.x, 10.x.x.x, 172.16-31.x.x) or external? This tells you the direction of traffic.

Step 2: Check the Protocol Column

TCP, UDP, ICMP, ARP, DNS, DHCP, OSPF? This immediately tells you what kind of communication is happening.

Step 3: Read the Info Column

This is the summary. For TCP, look at the flags in brackets: [SYN], [ACK], [RST]. For ICMP, look for Echo request or Destination unreachable. For DNS, look for the queried domain name.

Step 4: Expand Packet Details If Shown

The exam may show expanded headers. Look for:

  • TTL — how many hops left (low TTL = many hops or near expiry)
  • Window Size — receiver's buffer capacity (0 = flow control issue)
  • Seq/Ack numbers — track data flow and retransmissions
  • Src/Dst Port — identifies the application (80=HTTP, 443=HTTPS, 53=DNS, 22=SSH)
  • Src/Dst MAC — identifies the Layer 2 path (changes at each hop)

Troubleshooting Patterns

What to look for when something is broken.

  • No response at all
    tcp.flags.syn == 1 && !tcp.flags.ack
    You see SYN but no SYN-ACK. Firewall is dropping packets silently, host is down, or route doesn't exist. Check: is the SYN going to the right IP? Is the TTL too low?
  • Connection refused (RST)
    tcp.flags.reset == 1
    SYN goes out, RST comes back. Port is closed — no service listening. Check: wrong port number? Service not started?
  • Slow connection (retransmissions)
    tcp.analysis.retransmission
    Packets being resent. Packet loss on the link, congestion, or hardware errors. Check: how many retransmissions? Getting worse over time?
  • DNS failure
    dns
    Query goes out, no response (DNS server down) or response says NXDOMAIN (name doesn't exist). Check: is the DNS server IP correct? Is UDP 53 blocked?
  • DHCP not working
    dhcp
    You should see Discover → Offer → Request → Ack (DORA). If only Discover with no Offer: DHCP server is unreachable. If request but no ACK: address conflict.
  • Duplicate IP (ARP conflict)
    arp
    Two different MAC addresses responding to the same IP in ARP replies. Clear sign of IP conflict.
  • Routing loop
    icmp.type == 11
    ICMP Time Exceeded messages from routers. Packet is bouncing between routers until TTL hits 0. Check routing tables.
  • MTU / fragmentation issue
    ip.flags.mf == 1 || icmp.type == 3 && icmp.code == 4
    Fragmented packets or "fragmentation needed but DF set" ICMP messages. Reduce MTU or enable Path MTU Discovery.

CCNA Exam Tips

What Cisco Actually Tests

  • Reading a capture and identifying what protocol is running
  • Identifying the TCP 3-way handshake in a capture
  • Matching port numbers to services (80, 443, 22, 23, 53, 67/68, 25, 110, 143)
  • Recognizing ARP request/reply pairs
  • Identifying DHCP DORA process
  • Understanding why a connection failed based on packet evidence
  • Knowing the difference between TCP and UDP behavior in a capture
  • Identifying source/destination at each OSI layer (MAC changes per hop, IP stays same)
