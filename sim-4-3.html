DHCP DORA + DNS Lookup Sim

A fresh client knows nothing — no IP, no gateway, no DNS. DHCP gives it identity. Then it wants `www.example.com` but the network only routes IPs. DNS turns names into addresses. Two protocols, both critical, both touched on every reboot.

DHCP — DORA Handshake (UDP 67/68)

1. Discover Client broadcasts to 255.255.255.255 src=0.0.0.0 dst=255.255.255.255 — "anyone home?"
2. Offer Server replies with proposed IP, mask, gateway, DNS, lease time
3. Request Client broadcasts back: "I want the offer from server X" (so other servers know)
4. Acknowledge Server confirms. Client now has 192.168.1.50/24, GW 192.168.1.1, DNS 8.8.8.8, lease 24h.

DNS Resolution (UDP/TCP 53)

1. Stub resolver Client OS asks configured resolver (8.8.8.8): "A record for www.example.com?"
2. Recursive → root Resolver hits a root server: "who handles .com?"
3. Root → TLD Root replies: "go ask the .com TLD nameservers"
4. TLD → Authoritative .com replies: "ns1.example.com is authoritative for example.com"
5. Authoritative → answer ns1.example.com replies: A = 93.184.216.34
6. Cached return Resolver caches per TTL, returns to client. Client opens TCP to 93.184.216.34:443.

DNS Record Types

A hostname → IPv4. www → 93.184.216.34
AAAA hostname → IPv6. www → 2606:2800::
CNAME alias → another name. blog → www.example.com
MX mail server. example.com → mail.example.com pri 10
NS nameserver authority for the zone
PTR reverse: IP → hostname (in-addr.arpa)
TXT arbitrary text — SPF, DKIM, domain verification
SOA zone metadata — primary NS, serial, TTLs
