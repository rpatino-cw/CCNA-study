Three Planes, Three Jobsdrill

Every router/switch has three planes running side-by-side: control (decisions: routing protocols, ARP, STP), data (forwarding: CEF, TCAM lookup, send-the-packet), management (humans + tools: SSH, SNMP, NETCONF). Traditional networks: every device runs all three. SDN: control plane lifts onto a controller, data plane stays on the box. Drop the chips into the right lane.

The three planes (1-second mental model)

Control plane

How the device decides. Builds the routing table, the MAC table, STP topology. Slow path: CPU-driven. Examples: OSPF, BGP, ARP learning, STP BPDU calc.

Data plane

How the device forwards. Takes a packet, looks up the FIB / MAC table, ships it out. Fast path: ASIC / TCAM. Per-packet, line-rate. Examples: CEF FIB lookup, dot1q tag insert, ACL match in hardware.

Management plane

How you talk to the box. Out-of-band config + observability. Examples: SSH, SNMP polling, syslog, NETCONF, RESTCONF, console.

Traditional vs SDN — what moves

Mental shortcut: ask "where does the FIB live?" Traditional: on the device. SDN: still on the device — but built by a controller, not by OSPF running locally. Data plane never moves; control plane is what gets centralized.

Worked example — packet life on a traditional router

  1. Boot → IOS starts OSPF process. Control plane exchanges hellos, builds LSDB, runs SPF, installs best routes into RIB.
  2. RIB feeds FIB in TCAM. Data plane now has hardware-ready forwarding table.
  3. Packet arrives → ASIC does longest-prefix-match in TCAM → rewrites L2 header → ships out the egress port. Data plane only.
  4. Operator types show ip route over SSH. Management plane reads the RIB and prints it.

Faded prompt — your turn

A vendor pitches "we centralized the control plane." Translate: routing decisions are computed off-box and pushed down. The forwarding ASIC stays. CLI per box may go away (replaced by API). Management plane shifts toward the controller too.

Counter-question worth asking: "What happens to data forwarding if the controller dies?" Different vendors handle this differently — Cisco SD-Access keeps cached forwarding alive (graceful failure). Pure OpenFlow with no fallback drops everything not in the flow table.
