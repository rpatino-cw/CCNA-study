Two Networks Stackeddrill

Modern fabrics run two networks at once. The underlay is the boring physical IP network — switches, routed /31s, IS-IS or OSPF moving packets between them. The overlay is virtual tunnels (VXLAN) running on top of the underlay. Tenant traffic lives in the overlay. The fabric is the whole system — underlay + overlay + control plane — managed as one. Two-network mental model

Underlay (the wires + IPs)

Plain IP routing between switches. Spine-leaf with /31 routed links. IS-IS or OSPF finds shortest paths. Underlay forwarding is dumb and stable: "given dst IP X, send to next-hop." It does not know about tenants, VLANs, or services.

Overlay (the tunnels)

VXLAN tunnels between leaf switches (VTEPs). Each tenant/VLAN gets a 24-bit VNI (16M values vs 4096 VLAN IDs). Original Ethernet frame is wrapped in IP/UDP and dropped on the underlay. Decapsulated at egress VTEP. Same VNI on both ends = same logical L2 segment.

Fabric = both, plus a control plane (BGP-EVPN typically) coordinating who-owns-which-MAC + which VTEP terminates each VNI. A controller (Cisco DNA Center / ACI APIC) sits above and pushes intent.

Worked example — packet life across a VXLAN fabric

! Host A (10.10.10.5, VLAN 10) sends to Host B (10.10.10.9, also VLAN 10) — different leaf switches. Step 1 — at ingress leaf VTEP-1: Original frame: [eth-hdr | VLAN 10 | IP A → IP B | payload] Lookup: VLAN 10 maps to VNI 10010. Remote VTEP for that MAC = VTEP-2 (10.0.0.2). Step 2 — VXLAN encapsulation at VTEP-1: Outer L2 hdr | Outer IP src=10.0.0.1 dst=10.0.0.2 | UDP dst=4789 | VXLAN VNI=10010 | [original frame] Pushed onto the underlay. Step 3 — underlay forwards (IS-IS / OSPF) outer-IP from leaf 1 to spine to leaf 2: Underlay routers see only 10.0.0.1 → 10.0.0.2. They don't care about VNI 10010 or the inner frame. Step 4 — at egress VTEP-2: Strip outer L2/IP/UDP/VXLAN header → recover original Ethernet frame → put on VLAN 10 → forward to Host B. ! Host A and Host B think they're on one big L2 segment. Underlay never knew tenant data existed.

Faded prompt — your turn

A spine-leaf link goes down between Spine-1 and Leaf-3. Tenant traffic carrying VXLAN — does it break?

Answer: Probably no — if there's another spine and the underlay reroutes (ECMP / SPF). The overlay tunnel is logical; it doesn't care which physical path the underlay picks. Underlay convergence happens (IS-IS/OSPF), traffic shifts to the surviving spine, VXLAN keeps tunneling. The whole point of separating underlay from overlay is this: physical change → underlay reconverges → overlay invisible to tenants.

Counter-case: if Leaf-3 itself dies, the VTEP for that leaf's hosts is gone — those hosts are unreachable until the leaf is back. Overlay can't paper over a dead endpoint.
