How to Troubleshoot

Troubleshooting is not guessing. It is a repeatable loop: pin the symptom, read the device with the right show command, form one hypothesis, test it, fix, verify. This page teaches the loop, then gives you a symptom → check → cause → fix card for every P1 and P2 pillar. On the exam these are the drag-the-fault and "what command confirms this" questions.

The loop

Seven steps, every time

Run them in order. Most failed troubleshooting skips step 1 (define) or step 7 (verify) and chases the wrong fault.

1

Define

What exactly is broken, for whom, since when? One host or all? Pin the scope.

2

Gather facts

Read the device. show commands and counters, not assumptions.

3

List causes

What could produce this symptom? Rank by likelihood and by how fast you can test.

4

Hypothesis

Pick ONE cause to test first. Change one thing at a time.

5

Test

Prove or kill the hypothesis. If wrong, back out the change, return to step 4.

6

Fix

Apply the smallest change that resolves the root cause, not the symptom.

7

Verify + document

Re-test end to end. Confirm nothing else broke. Note what it was.

Pick a direction

Four ways to attack the OSI stack

The loop tells you how. The approach tells you where to start. Choosing the right one is the difference between 2 minutes and 20.

Bottom-up

When: many users down, hardware suspect

Start at L1 (cable, light, link) and climb. Slow but thorough. Best when the physical layer is in doubt or a whole segment is dark.

Top-down

When: one app fails, infra looks healthy

Start at L7 (the app) and descend. Best when ping works but a service does not, so the lower layers are clearly fine.

Divide-and-conquer

When: you have a hunch

Jump straight to L3: ping and traceroute. If L3 works, go up; if it fails, go down. Fastest in practice and the default on the exam.

Follow-the-path / swap

When: intermittent or one-link

Trace source to destination hop by hop, or compare a working port/device against the broken one and spot the single difference.

Reflexes

The first-60-seconds commands

Before you theorize, type these. They answer "is it up, is it addressed, can it reach, who is connected" for almost any fault.

show ip interface briefAre interfaces up/up? up/down = L2 problem, admin down = shutdown, down/down = L1.
show interfaces statusPort state, speed, duplex, VLAN, and err-disabled in one view.
ping / tracerouteL3 reachability and where in the path it dies.
show ip routeIs there a route to the destination at all? Longest match, then lowest AD.
show cdp neighborsWho is physically connected to which local port (confirms topology).
show loggingThe device often already told you what broke (err-disable, flaps, SLA).

Symptom shortcuts the exam loves: 169.254.x.x = DHCP failed (APIPA). Late collisions + slow link = duplex mismatch. Link down = speed mismatch. OSPF stuck in Exstart/Exchange = MTU mismatch. Port suddenly err-disabled = BPDU Guard or port-security violation.
