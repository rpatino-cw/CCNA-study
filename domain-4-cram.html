NAT / PAT

Inside Local = private, Inside Global = public. Static = manual 1:1, Dynamic = pool 1:1 (exhausts), PAT/overload = many-to-1 via ports.

▶ Start
Mark inside interfaceinterface g0/1 ip nat inside
Mark outside interfaceinterface g0/0 ip nat outside
Static 1:1ip nat inside source static 192.168.1.10 203.0.113.10
Dynamic poolip nat pool POOL1 203.0.113.1 203.0.113.6 netmask 255.255.255.248
PAT via poolip nat inside source list 1 pool POOL1 overload
PAT via outside interfaceip nat inside source list 1 interface g0/0 overload
🔍 Verify
show ip nat translations→ active mappings: Inside local -> Inside global (port col only filled for PAT)
show ip nat statistics→ hit count, which ACL + pool is active, total translations
clear ip nat translation *→ wipes DYNAMIC entries only; static entries stay
show running-config | include ip nat→ ip nat inside/outside on right interfaces + source line
❓ Likely Q+A
Which NAT type best conserves public IPv4?
PAT (NAT Overload). One public IP serves many hosts via unique port numbers (~65k ports).
What is the Inside Global address?
The PUBLIC IP after translation, the address the outside world sees. Inside Local = the private pre-NAT IP.
Dynamic NAT pool is exhausted. What happens to the next packet?
It is dropped. No overload means no port multiplexing, so when all pool IPs are used new flows fail.
A NAT ACL denies traffic. Is it dropped?
No. Denied just means not translated. The packet still routes (or not) per the routing table. ACL identifies, it does not filter.
What is the only config difference between dynamic NAT and PAT?
The overload keyword on the ip nat inside source line.
✅ Working when
  • show ip nat translations lists active mappings; a port number in the Inside global column proves PAT is working
  • show ip nat statistics shows the hit count climbing and names the active ACL + pool
  • A ping/curl from an inside host to an outside address succeeds and a return entry appears in the translation table
  • show run confirms ip nat inside on the LAN interface and ip nat outside on the WAN interface (missing one = NAT silently does nothing)
🧠 In-Out is WHERE, Local-Global is VIEW. Local=private, Global=public. Overload = many share one.
⚠️ Trap: The #1 exam trap: forgetting ip nat inside / ip nat outside on the interfaces, so NAT silently does nothing. Second trap: a NAT ACL deny is NOT a drop, it just skips translation. Third: clear ip nat translation * removes only dynamic entries, never static.
💀 Hardest
R1 has Fa0/0 (LAN, ip nat inside) and S0/0 (WAN, ip nat outside, IP 203.0.113.1/30). Config: access-list 1 permit 192.168.1.0 0.0.0.255 and ip nat inside source list 1 interface S0/0 overload. Host A (192.168.1.10) and Host B (192.168.1.11) both browse to server 8.8.8.8 at the same time. Give the four NAT addresses (inside local, inside global, outside local, outside global) for Host A's flow, and explain exactly how the router keeps A's and B's return traffic separate when both map to the same inside global IP.
Host A flow: inside local = 192.168.1.10 (A's real private IP), inside global = 203.0.113.1 (R1's S0/0 interface IP, since interface overload uses the WAN IP), outside local = 8.8.8.8, outside global = 8.8.8.8 (same as outside local, since there is no destination NAT). Both A and B share the single inside global 203.0.113.1, so the router cannot tell them apart by IP alone. With overload (PAT) it also translates the Layer 4 source PORT. R1 builds full translation entries keyed on protocol + IP + port: e.g. A becomes 203.0.113.1:50001 and B becomes 203.0.113.1:50002 (router picks unique source ports if the originals would collide). The NAT table stores the tuple (inside local IP:port, inside global IP:port, outside IP:port). When 8.8.8.8 replies, the packet arrives at 203.0.113.1 with a DESTINATION port of 50001 or 50002. R1 looks up that port in the NAT table, finds the matching inside local IP:port, rewrites the destination back to 192.168.1.10:origport (or .11), and forwards it out Fa0/0. The port number is the demultiplexing key, that is the whole point of PAT and why one public IP serves thousands of hosts.

DHCP (server + relay)

IOS hands out IPs from a pool (DORA); ip helper-address forwards DHCP across a router; DHCP snooping blocks rogue servers.

▶ Start
Exclude statics first (do before pool)ip dhcp excluded-address 10.1.1.1 10.1.1.9
Make the poolip dhcp pool LAN
Subnet to leasenetwork 10.1.1.0 255.255.255.0
Default gatewaydefault-router 10.1.1.1
DNS serverdns-server 8.8.8.8
Lease time (days hrs min)lease 0 12 0
Relay: on client SVI/interface, point at serverip helper-address 10.2.2.10
🔍 Verify
show ip dhcp binding→ client MAC to IP leases handed out
show ip dhcp pool→ leased count, available range
show ip dhcp conflict→ addresses dropped on duplicate detection
show running-config | section dhcp→ pool, excluded ranges, options
show ip interface gi0/1→ Helper address is 10.2.2.10
❓ Likely Q+A
What does DORA stand for, step by step?
Discover (client broadcast), Offer (server offers IP), Request (client broadcasts request for that IP), Ack (server confirms lease).
Which DORA messages are broadcast?
Discover and Request are broadcast (client has no IP yet). Offer and Ack can be unicast or broadcast.
Server is on another subnet across a router. What command, and where?
ip helper-address SERVER-IP on the interface or SVI facing the CLIENTS (the side that receives the broadcast).
Why exclude addresses, and before or after the pool?
To protect statics (router, servers, printers). Set ip dhcp excluded-address globally, best done before defining the pool.
What UDP ports does DHCP use?
Server listens on UDP 67, client on UDP 68.
✅ Working when
  • show ip dhcp binding lists active leases by MAC and IP, proving DORA completed
  • Client got an address in the pool range and NOT in the excluded range
  • show ip interface confirms Helper address is set so the relay forwards across the router
  • Empty show ip dhcp conflict means no duplicate-IP problems
🧠 DORA crosses the river on a HELPER; 67 serves, 68 begs.
⚠️ Trap: ip helper-address goes on the CLIENT-side interface, not the server side. It relays UDP broadcasts (DHCP 67/68 plus a few others) as unicast to the server. Also: excluded-address is a GLOBAL command, not inside the pool, and the relay rewrites the giaddr field so the server picks the right pool.
💀 Hardest
A client on VLAN 10 (SVI 10.1.1.1/24, ip helper-address 10.5.5.10 set there) gets NO address. The DHCP server at 10.5.5.10 has a pool for 10.1.1.0/24, but show ip dhcp binding on the server is empty. The relay router can ping 10.5.5.10. What is the most likely fix, and why does giaddr matter?
The relay reaches the server, so the break is scope selection or the return path. When the relay forwards the Discover, it inserts its client-facing SVI IP (10.1.1.1) into the giaddr (gateway IP address) field. The server uses giaddr two ways: (1) to pick which scope matches (the 10.1.1.0/24 pool) and (2) as the unicast address to send the Offer back to. Most likely fix: the server has no route back to 10.1.1.0/24 (to giaddr 10.1.1.1), so its Offer is dropped; add a return route on the server, or if a firewall sits between, permit UDP 67 both directions. Rule out: helper pointed at the wrong (server-side) interface, or the server pool subnet not matching giaddr so no scope is found. CCNA point: giaddr is what lets one central server serve many remote subnets; it is both the scope selector and the unicast return address, so reachability back to giaddr is mandatory.

NTP (Network Time Protocol)

NTP syncs device clocks to a time source over UDP 123; lower stratum = closer to the source.

▶ Start
Point router at a serverntp server 10.1.1.1
Backup server (optional)ntp server 10.1.1.2
Make THIS device a server (master)ntp master 3
Auth: define keyntp authentication-key 1 md5 CISCO
Auth: trust + turn onntp trusted-key 1 ; ntp authenticate
Set timezone firstclock timezone CST -6
🔍 Verify
show ntp status→ Clock is synchronized, stratum N, ref to server IP
show ntp associations→ * = synced peer, + = candidate; check st (stratum)
show clock→ leading * means clock NOT yet authoritative/synced
show ntp associations detail→ configured, our_master, sane, valid
❓ Likely Q+A
What transport + port does NTP use?
UDP 123.
Stratum 0 vs 1 vs 16?
0 = reference clock (atomic/GPS, off-network). 1 = directly attached to it. 16 = unsynchronized/invalid.
Lower or higher stratum is closer to source?
LOWER. Each hop away from the source adds one to the stratum.
Command to set a device as its own time source?
ntp master <stratum>. Plain 'ntp master' defaults to stratum 8.
Why does accurate time matter?
Log timestamps align across devices, certificate validity windows, and Kerberos tickets (~5 min skew or auth fails).
✅ Working when
  • show ntp status reads 'Clock is synchronized' and shows a stratum below 16.
  • show ntp associations shows a * next to the chosen server and a sane stratum (st).
  • show clock no longer has the leading * (asterisk = unsynced/unreliable).
🧠 Low stratum, close to source. UDP 123, sync or bust.
⚠️ Trap: 'ntp master' with no number defaults to stratum 8, not 1. In show clock the leading asterisk (*) means time is NOT trusted yet; people read it backwards.
💀 Hardest
R1 has 'ntp server 10.1.1.1' and 'ntp master 5'. show ntp status reads 'Clock is synchronized, stratum 6, reference 10.1.1.1'. The link to 10.1.1.1 then fails. What stratum does R1 advertise, and why doesn't it just go unsynchronized (stratum 16)?
R1 falls back to its own configured master and advertises stratum 5, now sourcing time from its internal hardware clock (ref 127.127.x.x) instead of stratum 6 via 10.1.1.1. It does NOT drop to stratum 16 because 'ntp master 5' makes R1 an authoritative fallback source: while the external server is reachable R1 prefers it (the server at stratum 5 gives R1 stratum 6), but on failure R1 keeps serving time from its local clock at the configured master stratum. Key idea: ntp master is a survivability fallback so downstream clients never lose a source, but its time is only as good as R1's local oscillator (drifts, no real reference). CCNA trap: clients keep syncing happily and never alert, even though the time may now be wrong.

DNS

DNS maps names to IPs (UDP/TCP 53); IOS can be a DNS client (resolver) or a DNS server.

▶ Start
Set DNS server(s) client usesip name-server 8.8.8.8 8.8.4.4
Turn name resolution ON (default)ip domain-lookup
Append domain to short namesip domain name lab.local
Static name to IP (no server needed)ip host R2 10.0.0.2
Make IOS a DNS serverip dns server
🔍 Verify
show hosts→ Name server list + cached/static name-to-IP entries
show running-config | include name-server|domain→ name-server IPs and domain name lines present
ping R2→ Name resolves to IP, then ICMP replies (proves lookup worked)
❓ Likely Q+A
Default state of ip domain-lookup?
Enabled. A mistyped command triggers a DNS lookup + telnet attempt (the hang). Fix: no ip domain-lookup.
What port does DNS use?
UDP 53 for queries; TCP 53 for zone transfers / large responses.
Command to list the DNS servers the device queries?
ip name-server (up to 6). Verify with show hosts or show run.
DNS client vs DNS server on IOS?
Client (resolver): uses ip name-server to ask others. Server: ip dns server, answers from its host table/cache.
Old vs new domain command?
New: ip domain name X. Older IOS: ip domain-name X (hyphen). Know both.
✅ Working when
  • show hosts lists name servers and resolved entries
  • ping/telnet by hostname succeeds (name resolved to IP)
  • show run shows ip name-server and ip domain-lookup config
🧠 "Name-server asks, domain-lookup allows, ip host hardcodes, dns server answers."
⚠️ Trap: Do not confuse ip name-server (who I ask) with ip dns server (I become the answerer). domain-lookup is ON by default, so a typo at the prompt hangs trying to resolve it.
💀 Hardest
R1 has no ip name-server set and ip domain-lookup is still on. An admin types badcmd at the exec prompt and the session freezes ~30s. Explain exactly why, what R1 tries, and two cleanest fixes.
R1 cannot parse badcmd as a command, so with ip domain-lookup enabled it treats it as a hostname to resolve. With NO ip name-server configured, IOS falls back to broadcast (255.255.255.255) DNS queries, gets no answer, and the resolver times out (the ~30s hang) before it would Telnet to the resolved host. Fixes: (1) no ip domain-lookup to disable name resolution globally, or (2) transport preferred none on the line to stop the auto-Telnet-on-typo behavior. Escape mid-hang with Ctrl-Shift-6. Key facts: lookup is on by default; DNS is UDP 53; with no name-server IOS broadcasts queries rather than failing instantly.

SNMP + Syslog

SNMP = NMS polls/sets agents (v2c community = plaintext, v3 = auth+priv); Syslog = severity 0-7, sent to console/buffer/server UDP 514.

▶ Start
SNMP read communitysnmp-server community READ ro
SNMP v3 user (auth+priv)snmp-server user USER GRP v3 auth sha PASS priv aes 128 KEY
Send traps to NMSsnmp-server host 10.1.1.1 version 3 priv USER
Enable trapssnmp-server enable traps
Syslog to serverlogging host 10.1.1.2
Syslog severity filter (<=4)logging trap warnings
Log to bufferlogging buffered 16384
🔍 Verify
show snmp→ chassis, contact, packets in/out
show snmp user→ v3 user, auth SHA, priv AES
show snmp community→ RO/RW string + access list
show logging→ trap level, logging host(s), buffer contents
❓ Likely Q+A
What UDP port does syslog use?
514. (SNMP uses 161 for get/set, 162 for trap/inform.)
v2c vs v3 security?
v2c = community string in cleartext, no encryption. v3 adds authentication (MD5/SHA) and privacy/encryption (DES/AES). v3 = the only secure version.
Trap vs Inform?
Both agent-to-NMS alerts. Trap = fire-and-forget (no ack, UDP 162). Inform = acknowledged + retransmitted if no ack. Inform = reliable.
Name severity 0 and 7.
0 = Emergency (system unusable). 7 = Debugging (most verbose). Lower number = more severe.
logging trap 4 means what gets logged?
Levels 0-4 (Emergency through Warning). Filters are <= the configured level; higher-numbered (less severe) are dropped.
✅ Working when
  • show snmp user shows the v3 user with auth SHA and priv AES = security model active
  • NMS receives traps/informs and show snmp counters increment on the agent
  • show logging lists the logging host and the trap level, and buffer shows recent events
  • syslog server on UDP 514 receives messages stamped with the facility.severity
🧠 Syslog 0-7: Every Awesome Cisco Engineer Will Need Ice-cream Daily (Emergency, Alert, Critical, Error, Warning, Notification, Informational, Debugging).
⚠️ Trap: SNMP get/set = UDP 161, trap/inform = UDP 162; syslog = UDP 514. Do not mix them up. Also: severity is INVERSE, level 0 is the most severe (Emergency), level 7 is least (Debugging). logging trap warnings = 0 through 4.
💀 Hardest
An NMS at 10.0.0.5 gets SNMPv3 informs but no config changes succeed, and the syslog server sees nothing below level 5. The agent has: snmp-server group G v3 priv read RO-VIEW; snmp-server user U G v3 auth sha P1 priv aes 128 P2; snmp-server host 10.0.0.5 informs version 3 priv U; logging host 10.0.0.5; logging trap notifications. Explain both failures and fix each.
FAILURE 1 (no sets): the group G is defined with only a 'read RO-VIEW' clause and no 'write' view. Without a write view the v3 user can poll (get) and receive notifications but every set is denied, even with full auth+priv. Informs still arrive because notifications use the notify path, not the write view. Fix: add a write view, e.g. snmp-server view RW-VIEW iso included, then snmp-server group G v3 priv read RO-VIEW write RW-VIEW. FAILURE 2 (no syslog below 5): logging trap notifications sets the severity threshold to level 5 (Notification). Syslog filtering is <= the configured level, so only 0-5 are sent; levels 6 (Informational) and 7 (Debugging) are dropped, and that is expected. If the goal is to see Informational/debug events on the server, raise the threshold: logging trap informational (level 6) or logging trap debugging (level 7). Key traps: write view governs SNMP sets independent of auth/priv; syslog level names map to numbers (notifications=5, informational=6, debugging=7) and the filter is cumulative from 0 up to that number. Both knobs are about scope/authorization, not the v3 crypto, which was already correct.

QoS (classification, marking, queuing, shaping vs policing)

QoS classifies and marks traffic, then queues it so voice gets low-latency priority while bulk traffic absorbs delay and drops.

▶ Start
Mark voice DSCP EF (46)set dscp ef (in policy-map class voice)
Mark call-signaling CS3set dscp cs3
Give voice strict priority queuepriority percent 10 (or priority 128)
Trust phone CoS at access portmls qos trust cos / trust device cisco-phone
Shape egress (buffer, no drop)shape average 10000000
Police ingress (drop/remark over rate)police 8000000 conform-action transmit exceed-action drop
🔍 Verify
show policy-map interface g0/1→ per-class match/drop counts, priority queue, shape/police rates
show mls qos interface g0/1→ trust state (trust cos/dscp) and trust boundary
show mls qos maps cos-dscp→ default cos-dscp map: CoS 5 -> DSCP 40 (CS5), not EF. Remap to 46 (EF) for voice.
show queueing interface g0/1→ queue strategy, priority queue active
❓ Likely Q+A
DSCP value for voice (RTP)?
EF = 46. Call signaling = CS3 (24).
Shaping vs policing in one line?
Shape buffers excess (smooths, adds delay), police drops or remarks excess (no buffer).
Where is the layer-2 marking and the layer-3 marking?
CoS = 3 bits in the 802.1Q tag (L2). DSCP = 6 bits in the IP ToS/DS byte (L3).
What is the trust boundary?
The point where you start believing incoming markings. Push it as close to the source as possible, usually the IP phone, not the PC behind it.
Why does voice need a priority queue?
Voice is delay and jitter sensitive and cannot retransmit. LLQ gives it strict priority so it is served first, capped to protect other traffic.
✅ Working when
  • show policy-map interface shows packets matched and dropped per class, priority queue counters rising for voice
  • show mls qos interface confirms the port trusts CoS/DSCP at the trust boundary
  • Police drop counters climb under load; shape shows delayed/buffered packets not drops
  • Voice MOS stays high (no jitter/loss) while bulk traffic queues during congestion
🧠 Police Drops, Shape Saves (buffers). EF=voice=46.
⚠️ Trap: CoS lives in the 802.1Q tag, so it is LOST on an access (untagged) link, only survives on a trunk. DSCP is in the IP header and survives end-to-end. Also: shaping buffers (adds delay), policing drops, do not swap them. Trust the phone, NOT the PC.
💀 Hardest
A trunk carries voice marked CoS 5 / DSCP EF and bulk data. The egress WAN link is 10 Mbps but the provider polices at 8 Mbps and drops anything over. You configure shape average 8000000 with an LLQ giving voice priority percent 10. Explain (a) why shaping here is correct and policing on your side would be wrong, (b) what happens to voice if you only shape but forget the priority queue, and (c) why the CoS 5 marking alone will not protect voice once traffic crosses an L3 router hop.
(a) The provider POLICES at 8 Mbps and drops excess, so you must SHAPE your egress to 8 Mbps to stay under their rate. Shaping buffers bursts above 8 Mbps and meters them out smoothly, avoiding the provider drops. If you policed on your own side you would just drop your own traffic instead of smoothing it, throwing away packets you could have queued. (b) Shaping alone creates a buffer/queue. Without a priority (LLQ) class, voice packets sit in the same FIFO/CBWFQ queue as bulk data behind the shaper. During congestion that adds queuing delay and jitter, RTP cannot retransmit, so call quality degrades even though no policer is dropping. The priority queue is what serves voice FIRST out of the shaper buffer with bounded latency. (c) CoS (802.1Q PCP, 3 bits) is a LAYER 2 field that exists only inside the VLAN tag on a trunk. The moment the frame is routed at L3 the 802.1Q header is stripped and rebuilt, so CoS is lost unless it was mapped into DSCP. DSCP (6 bits in the IP DS byte) is layer 3 and survives every hop. Correct design: trust/mark at the access edge, map CoS 5 to DSCP EF (46), and key all downstream queuing on DSCP EF so voice stays protected across router hops, not just on the local trunk.
