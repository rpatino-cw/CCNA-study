1. The 30-Second Answer read this twice

A firewall enforces a security policy at a network boundary. NGFW = stateful firewall + deep packet inspection + application-ID + user-ID + IPS + URL filter + threat intel. IPS = inline scanner that blocks. IDS = passive scanner that alerts.
  • Stateless firewall (ACL-style) = matches packets on 5-tuple only (src/dst IP, src/dst port, protocol). No connection memory.
  • Stateful firewall = tracks connections in a state table. Return traffic for an established connection is allowed automatically.
  • NGFW = stateful + Layer-7 awareness. Sees that "TCP/443" is actually Facebook or Salesforce, can block by app/user, decrypts TLS, runs IPS, fetches Talos threat feeds.
  • IDS = sits OUT-of-band on a SPAN/TAP, alerts on bad traffic. IPS = sits IN-line, drops bad traffic.

Memorize: NGFW = stateful + L7. IDS sees. IPS blocks.

2. NGFW Anatomy at a Glance visual

NGFW stacks: stateful conn tracking (table) + DPI (App-ID, User-ID) + IPS (Snort signatures) + URL filtering + TLS decrypt + VPN termination + cloud threat intel (Talos). Zones (inside/outside/DMZ) define trust boundaries; policies match on zone+app+user.

3. Packet Flow Through an NGFW memorize

1
Ingress on a zone interface (e.g. outside). Read 5-tuple: src IP, dst IP, src port, dst port, protocol.
2
Connection lookup. Match against state table. If existing flow + stateful match → fast-path forward.
3
Policy lookup for new flows: source zone + dest zone + 5-tuple + (NGFW only: App-ID, User-ID, URL category). Default = deny.
4
NAT applied if rule matches (PAT for outbound, static NAT for inbound DMZ).
5
TLS decrypt (NGFW with decrypt policy) — terminate SSL with FW's own cert, inspect cleartext.
6
IPS scan — signatures match against payload. Drop / alert / log per rule action.
7
Re-encrypt + forward out the egress interface. Update state table. Log to mgmt.
Stateless firewall stops at step 1+3. Stateful firewall does 1-4. NGFW does 1-7.

4. Stateless vs Stateful vs NGFW 3-way compare

Stateless (ACL)

  • Matches per-packet only
  • 5-tuple match
  • NO connection memory
  • Return traffic needs explicit reverse rule
  • Fast, cheap, dumb
  • Cisco: router ACLs, pre-2000 ASA

Stateful firewall

  • Tracks connections (state table)
  • Auto-allows return traffic for established flows
  • L4 only (port + protocol)
  • Doesn't see app inside the port
  • Cisco: classic ASA

NGFW (next-gen)

  • Stateful + L7 awareness
  • App-ID — sees app, not just port
  • User-ID — policy by AD user, not just IP
  • Built-in IPS, URL filter, malware
  • TLS decrypt + threat intel
  • Cisco: Firepower / FTD / Secure Firewall

5-tuple = src IP + dst IP + src port + dst port + protocol. The fingerprint of a connection. Stateful firewalls hash this to track flows.

Trap to recognize
"App-ID = port number" — WRONG. App-ID inspects payload to identify the actual application. Many apps run on TCP/443 (HTTPS) — App-ID tells Salesforce from Facebook from a custom tunnel.

5. IDS vs IPS — The Most-Tested Compare memorize cold

AspectIDS (Detection)IPS (Prevention)
PositionOut-of-band (mirror / SPAN / TAP)In-line (traffic passes through it)
ActionAlerts only — cannot dropDrops / blocks / resets in real time
Latency impactNone (sees a copy)Adds inspection latency
Failure modeLoses visibility (traffic still flows)Risks dropping traffic if it fails — needs fail-open or HA
False positive costAnnoying alertBLOCKS legitimate traffic
Cisco productSnort in IDS mode (rare)FTD with IPS policy / Snort inline

Detection methods — both IDS and IPS use the same 3:

  • Signature-based — match known bad patterns (CVE exploit signatures, malware hashes). Fast, accurate on known threats. Misses zero-days.
  • Anomaly-based — baseline normal traffic, flag deviations. Catches new attacks. False positives in volatile environments.
  • Policy-based — admin defines what "bad" looks like (e.g. "block all traffic to TOR exit nodes"). Most static.
"IDS sees, IPS blocks." Position decides function: SPAN port = passive = IDS. Inline = active = IPS.

9. Cisco's Favorite Traps spot the wrong answer

Trap #1 — "IDS blocks bad traffic"
Wrong. IDS only alerts (out-of-band, mirror/SPAN). To BLOCK you need IPS (inline). Easy 1-point gimme if you know the position rule.
Trap #2 — "Stateful firewall = NGFW"
Wrong. Stateful = L4 only. NGFW adds L7 (App-ID, User-ID, IPS, URL filter). Classic ASA was stateful but not NGFW.
Trap #3 — "Firewall blocks at L1/L2"
Wrong. Firewalls operate at L3-L7. They forward by IP (or zone), inspect L4 ports, and (NGFW) parse L7 application data. They do NOT block by MAC by default (that's switch port-security).
Trap #4 — "TLS-encrypted traffic can't be inspected"
Half-wrong. NGFW can decrypt TLS with a decrypt policy + the FW's CA cert pushed to clients. Without decrypt, only TLS metadata (SNI, cert info) is visible. Inspection drops to "encrypted bypass" mode.
Trap #5 — "All Cisco ASAs are NGFWs"
Wrong. Classic ASA = stateful FW + VPN only. NGFW features (IPS, URL filter, App-ID) come from FirePOWER service module bolted on, OR by reimaging to FTD. Plain ASA without FirePOWER is NOT an NGFW.
Trap #6 — "Anomaly detection has zero false positives"
Wrong. Anomaly = baseline-driven — high false positive rate in volatile networks. Signature has fewer FPs but misses zero-days. Each method has tradeoffs.
Trap #7 — "Default deny vs default permit on Cisco firewalls"
Cisco firewalls (ASA / FTD) default to DENY for traffic from a low security zone to high (e.g. outside → inside). Higher to lower (inside → outside) is permitted by default unless restricted. Don't confuse with router ACLs which require explicit permits everywhere.

10. Top 10 Must-Know if-only-this

Memorize until automatic
  1. Firewall = L3-L7 boundary enforcer. Stateless = ACL. Stateful = tracks connections. NGFW = stateful + L7 (App-ID, User-ID, IPS, URL filter).
  2. IDS = sees, IPS = blocks. IDS is out-of-band (mirror/SPAN). IPS is inline. Position decides function.
  3. 5-tuple = src IP, dst IP, src port, dst port, protocol. The connection fingerprint stateful firewalls track.
  4. App-ID identifies application from payload, not port. Distinguishes Facebook from Salesforce both on TCP/443.
  5. NGFW packet flow: ingress → conn lookup → policy → NAT → TLS decrypt → IPS scan → re-encrypt + forward.
  6. Detection methods: signature, anomaly, policy. Signature = known bad patterns. Anomaly = deviation from baseline.
  7. Cisco NGFW lineup: FTD / Firepower / Secure Firewall (NGFW). Classic ASA = stateful only (NGFW via FirePOWER module).
  8. Snort = IPS engine. Talos = threat-intel feed. Both Cisco-owned (Sourcefire acquisition).
  9. ASA security levels: inside=100, outside=0, DMZ=50. Higher → lower allowed; lower → higher denied unless ACL permits.
  10. show conn = state table. show xlate = NAT translations. show access-list = rule hits. Core ASA/FTD verification commands.

11. Self-Check no peeking

Q1. A device sits on a SPAN port and alerts on suspicious traffic but cannot drop it. What is it?

Reveal

IDS (Intrusion Detection System). Out-of-band, passive. Cannot block. Refer back: §5, §9 trap #1.

Q2. Two stateful firewalls receive identical TCP/443 traffic. One blocks Facebook but allows Salesforce. What feature enables that distinction?

Reveal

App-ID (NGFW feature). Without it, both are just "TCP/443" and the firewall can't tell them apart. Refer back: §4.

Q3. A user from "outside" zone tries to reach a server in "inside" zone. ASA defaults — allowed or denied?

Reveal

Denied by default. Lower-to-higher security level requires explicit ACL permit. Refer back: §6, §9 trap #7.

Q4. NGFW vs traditional stateful firewall — what's the new capability?

Reveal

L7 awareness. App-ID, User-ID, IPS, URL filtering, TLS decrypt, threat intel. Stateful firewall stops at L4. Refer back: §4.

Q5. Which command shows current connections in the firewall's state table?

Reveal

show conn on ASA/FTD. Each line = a tracked flow with 5-tuple, idle time, byte count, flags. Refer back: §7.
