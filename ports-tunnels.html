Ports → Tunnels

Every port on your computer is a door. Each protocol leaves through its own door, crosses the public internet, and arrives at the matching listening port on a server.

Why this matters for the exam

The CCNA expects instant recall on well-known ports. But knowing the number is only half the picture — you also need to see where the packet goes: source socket → NAT → destination IP → destination port → listening service. This visual gives you the full tunnel, end to end.

Key concepts

Port = door

Every active connection ties a local socket src_ip:src_port to a remote socket dst_ip:dst_port. The kernel routes incoming packets to whatever process is bound to that port.

Source vs destination port

The source port is usually random/ephemeral (range 49152–65535). The destination port is the well-known service number — 80, 443, 22, etc.

NAT translation

Your home router rewrites 192.168.1.42:51234 → 203.0.113.42:51234 on the way out, and reverses it on the way back. Without NAT, replies wouldn't find your private IP.

Listening port

The server side runs nginx, sshd, or named bound to a port. netstat -an | grep LISTEN shows what's bound on a host.

TCP vs UDP

HTTP / SSH / SMTP / MySQL / FTP all use TCP — three-way handshake, ordered delivery. DNS and NTP mostly use UDP — one-shot, no handshake, faster.

DNS comes first

Typing example.com never opens a connection by itself. The OS first sends a UDP query to port :53, gets back an IP, then opens the TCP connection to :80 or :443.
